# [アイデアの更新](https://laracasts.com/series/laravel-from-scratch-2026/episodes/40)

アイデアの手順について話しましょう ― それをモーダルで追加し、検証して保存、アイデアページで表示し、更新時にも同期した状態を保つ方法についてです。
このエピソードでは Alpine.js（クライアント側）と Laravel（サーバ側）を組み合わせた処理が出てきます。
確かにいくつか複雑なところはありますが、どれも理解しやすいものばかりです

> 手順はその詳細と完了状態から成り立ちます。

忘れないでください：これはただのテキストと終わったかどうかの状態の組み合わせです。

## モーダルに手順を追加する

リンクの UI を再利用して、小さな “新しい手順” の入力欄を追加ボタンを追加しました。
モーダルでは Alpine で新規手順の状態と手順の配列を管理します。 

入力欄と Alpine のルートはこのようになります（簡略化しています）：

```html
<div x-data="{ newStep: '', steps: [] }">
  <label for="new_step" class="label">Actionable Steps</label>

  <div class="flex gap-2">
    <input
      id="new_step"
      x-model="newStep"
      placeholder="What needs to be done?"
      class="input flex-1"
      autocomplete="off"
      spellcheck="false"
    />

    <button
      type="button"
      class="btn"
      :disabled="!newStep"
      @click="steps.push(newStep); newStep = ''"
    >
      Add
    </button>
  </div>

  <template x-if="steps.length">
    <div class="mt-3 space-y-2">
      <template x-for="(step, index) in steps" :key="index">
        <div class="flex items-center justify-between bg-gray-50 p-2 rounded">
          <span x-text="step"></span>
          <button type="button" @click="steps.splice(index,1)" class="text-sm text-red-500">Remove</button>
        </div>
      </template>
    </div>
  </template>
</div>
```

アプリ構築中に Alpine の状態を調査したいときは、Alpine DevTools という Chrome 拡張機能を使いましょう ― とても便利です。

## サーバ側：バリデーション

フォームが送信されたら、`steps` のバリデーションを行いましょう。
最初は文字列の配列として手順を扱いましたが、編集では各手順を、詳細と完了状態とオプションで id を持つオブジェクトとして扱いたくなるでしょう。

バリデーションルールの例（フォームリクエストを使用）：

```php
// app/Http/Requests/StoreIdeaRequest.php (excerpt)
public function rules()
{
    return [
        'title' => ['required', 'string', 'max:255'],
        'description' => ['required', 'string', 'min:4'],
        'steps' => ['nullable', 'array'],
        // 文字列として扱うか、:
        'steps.*' => ['string', 'max:255'],

        // もしくは、手順をオブジェクトとみなす（編集時）
        'steps.*.id' => ['nullable', 'integer'],
        'steps.*.description' => ['required', 'string', 'max:255'],
        'steps.*.completed' => ['required', 'boolean'],
    ];
}
```

ご自身のフォームで手順をどのように送信するかに合わせて選んでください。
編集時には、完了状態を保持し既存の各手順を識別できるように、オブジェクトとして送信することが多いです。

## 新規作成時に手順を保存する

アイデアの作成時には、送信された手順は配列に変換してリレーションを使って作成することができます：

```php
// app/Http/Controllers/IdeaController.php (store excerpt)
$attributes = $request->safe()->except('steps');
$idea = auth()->user()->ideas()->create($attributes);

$steps = $request->input('steps', []); // 文字列の配列かオブジェクトのリスト
if (!empty($steps)) {
    $rows = collect($steps)->map(fn ($s) => [
        'description' => is_array($s) ? $s['description'] : $s,
        'completed' => is_array($s) ? ($s['completed'] ?? false) : false,
    ])->toArray();

    $idea->steps()->createMany($rows);
}
```

createMany は便利です：連想配列の配列を渡すと Eloquent がインサートを行います。

## 更新時に手順を同期させる

作成時に比べて更新時はややこしいです：手順は追加も変更も削除もできます。
これにはいくつか選択肢があります：アップサート（id で対応づけて更新／追加を行う）で行う方法か、もしくはよりシンプルで多くのアプリで適用できる方法です：既存の手順を削除し、送信された手順で作り直します。
雑な方法ですが確実で、多くの場合この方がシンプルになります。

更新手順の例です（トランザクション内部）：

```php
DB::transaction(function () use ($idea, $validated) {
    $idea->update($validatedAttributes);

    // 既存の手順を削除し、信頼できる情報源であるフォームから作り直す
    $idea->steps()->delete();

    $rows = collect($validated['steps'] ?? [])->map(fn ($s) => [
        'description' => $s['description'],
        'completed' => (bool) ($s['completed'] ?? false),
    ])->toArray();

    $idea->steps()->createMany($rows);
});
```

全削除＋作り直しはシンプルで、ちょっとした更新／マージ時のバグが避けられます。
既存の手順の ID やリレーションを保存しておかなければいけない場合は、アップサートを検討してください：

```php
// アイデア―手順のアップサートの例（対応するユニーク制約が必要）
IdeaStep::upsert($rows, ['id'], ['description', 'completed']);
```

ですがアップサートではキーの選択に注意を払わなければいけません ― リストが小さい場合はかえって手間が増えます。

## フォームからオブジェクト型の手順を送信する

フォームを信頼できる情報源とするために、各手順をオブジェクト（詳細＋完了状態＋id）として送信しましょう。
HTML では各手順に hidden 入力をつけ、送信データを二次元配列にします。

例（モーダル内の各手順）：

```html
<div x-for="(step, index) in steps" :key="index">
  <input type="text" :name="`steps[${index}][description]`" x-model="step.description" />
  <input type="hidden" :name="`steps[${index}][completed]`" :value="step.completed ? '1' : '0'"/>
  <input type="hidden" :name="`steps[${index}][id]`" x-model="step.id" />
</div>
```

Alpine から新規手順を追加するときは、オブジェクトをプッシュします：

```javascript
steps.push({ id: null, description: newStep, completed: false });
newStep = '';
```

注意：新たに追加された手順には `id` がないため、サーバ側でそれを扱えるようにしておきましょう。

## 完了状態の切替（手順ごとの更新）

トグルボタンには手順ごとに小さなフォームを配置して、`completed` の状態を切り替えを行う手順の更新ルートへ PATCH を送ることができます。
シンプルにサーバで処理するようにすれば、複雑な JavaScript を用意する必要はありません。

ルート：

```php
// routes/web.php
Route::patch('/steps/{step}', [\App\Http\Controllers\StepController::class, 'update'])
    ->name('steps.update')
    ->middleware('auth');
```

コントローラ：

```php
// app/Http/Controllers/StepController.php
public function update(Request $request, Step $step)
{
    $this->authorize('update', $step); // implement a policy

    $step->update(['completed' => ! $step->completed]);

    return back();
}
```

閲覧ビューではトグルボタンを小さなフォームでラップして、このルートへ POST 送信を行います。

## デバッグのコツ

もしテストや手動でのリクエストで誤ったデータが表示されたら、dd() を使って検証済みのリクエストを確認し、サーバは実際に受け取ったものを見てみましょう。

```php
dd($request->validated());
```

よくある落とし穴：JavaScript で入れ子になった入力名を構築する際、誤ってインデックスやキーを上書きしてしまうことがあります。
新しい項目に対して一貫したインデックスを生成したり、アドホックなクライアントIDを割り当てたりすると便利です。
このレッスンでは、サーバ ID を持たない項目用にインデックスの代替値を生成することで、複数の新規項目が衝突するのを防ぎました。

## テストと整理

重要なフローにはブラウザ／機能テストを用意しましょう：手順を含んだアイデアの作成、手順の修正と同期、完了状態の切替などです。
`assertSee` や `assertRedirect`、それからフォームには `assertInputValue`（編集モーダルの初期値をチェックする）といった追加のアサーションが便利です。

画面が増えるのに合わせてテストも整理しなおしましょう：`CreateIdeaTest` と `UpdateIdeaTest` に分けると分かりやすく、保守しやすくなります。

## 補足とトレードオフ

- 更新時に手順を全削除＋作り直しする方法は、小さなリストにおいてはシンプルで確実です。非常に大きいリストの場合や、手順の ID や監査証跡を残さなければならない場合は、アップサートや、もっと用途に即した同期アルゴリズムを使いましょう。
- クライアント側からオブジェクト形式で手順を送信する方法が一番制御しやすいです：詳細と完了状態とid。この方法ではサーバでの同期手順も単純になります。
- より動的な UX（全画面更新なし）にしたければ、各手順のフォームを Alpine + fetch/AJAX に置き換え、必要なところだけ DOM を更新するようにします。ただしサーバ側でのバリデーションと保存は省いてはいけません。
- テストがあればリファクタリングを安全・確実に行えます ― バリデーション、デフォルト値、リダイレクト、そして手順の同期についてのテストを追加しましょう。

## まとめ

これで一連の流れが完成しました：モーダルで手順を追加し（Alpine を使って）、構造を持った配列としてそれを送信し、フォームリクエストでバリデーションし、新規作成時には createMany でそれらを作成、更新時には全削除と作り直し（好みに応じてアップサート）で同期させました。
このエピソードではこれまでより多くの JavaScript が出てきました ― おそらくそこが一番分かりにくかったでしょう ― しかし一旦フォームの送信データが出来てしまえばサーバ側は単純です。

必要であれば、保存ロジックをまとめた（画像の取り扱い、手順の同期、通知）コンパクトな “CreateIdeaAction” と “UpdateIdeaAction” を別々に用意できます。
こうすることでコントローラは小さく、テストはしやすくなります ― 必要ですか？
