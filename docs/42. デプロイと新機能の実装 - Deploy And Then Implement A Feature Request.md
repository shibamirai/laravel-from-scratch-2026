# [デプロイと新機能の実装](https://laracasts.com/series/laravel-from-scratch-2026/episodes/42)

さあ仕上げに参りましょう：
フォーマッタを実行し、テストを実施し、Forge にプッシュし、デプロイでのつまづきに対処し、それからちょっとした機能（アイデアの詳細のマークダウン対応）を実装します ― 手早くテストをしてから本番環境にホットデプロイします。
ワークフローの最後はこうやって締めくくるのが私の好みです：自動チェック、すべてOKとなったテスト、そしてスムーズなデプロイ。

## フォーマット、テスト、修正、プッシュ

まず最初にプロジェクトのフォーマッタを実行します。

```bash
composer run format
```

これによって多少のコードが変更されることが多いです（クロージャの簡略化、スタイルの修正などそんなものです）。
次にテスト一式を実行します ― Pest でも artisan のラッパーでもどちらでも構いません：

```bash
# Pest
./vendor/bin/pest

# または
php artisan test
```

テストに失敗したらエラーをよく読んで修正しましょう。
私のケースでは、ログイン／登録のテストでホームページにリダイレクトするようにしていましたが、今のアプリはアイデアのインデックスページに遷移することになっています。
なので正しいルートにリダイレクトするようコントローラを修正して、テストもそのように対応させました。

登録フローで施した修正はこのようなものです：

```php
// app/Http/Controllers/Auth/RegisteredUserController.php
return redirect()->route('ideas.index');
```

それからログインフローでは redirect intended を使用し、フォールバックとしてアイデアのインデックスページを指定します：

```php
// app/Http/Controllers/Auth/AuthenticatedSessionController.php
return redirect()->intended(route('ideas.index'));
```

そしてテストを修正して新しいルートに対応させます：

```php
// tests/Feature/Auth/LoginTest.php (excerpt)
$this->post(route('login'), $credentials)
     ->assertRedirect(route('ideas.index'));
```

テストを繰り返してすべてパスするようになってからコミットします：

```bash
git add .
git commit -m "Fix redirects: send users to ideas index"
git push
```

## Forge にプッシュしデプロイする

GitHub にプッシュすると、Forge が自動的にこれを検知し、デプロイ用のフックを実行します。
デプロイに失敗したら、Forge のデプロイログを確認しましょう ― 環境やビルドの問題であることが多いです。

私のときは１度マイグレーションエラーが発生しました：SQLite では受け入れられていたカラムのデフォルト値が MySQL では使えませんでした。
実行環境でも SQLite を使うように（このデモのために）、Forge の環境と設定を修正しました：

```env
DB_CONNECTION=sqlite
```

その後 Forge で手動デプロイを開始し、ビルド過程、アセットのコンパイル、そしてマイグレーションが実行されるところを見ます。
成功したらサイトを訪れ、変更を確認しましょう。

## 詳細のマークダウン対応を実装する

機能の要請：閲覧ページでアイデアの詳細をマークダウンとしてレンダリングしてください。
これに対して、アイデアモデルにシンプルなアクセサを実装し、フォーマットされた HTML を Blade 上でレンダリングするようにしました。

> アクセサ（アクセスメソッド）とはモデルのカスタムプロパティのことです ― 基本的には属性を変形させたりフォーマットするゲッターです。

モデルには `formatted_description` というアクセサを追加しました。
これがその実装です：

```php
// app/Models/Idea.php

use Illuminate\Support\Str;

class Idea extends Model
{
    // ...

    public function getFormattedDescriptionAttribute()
    {
        return Str::of($this->description)->markdown();
    }
}
```

閲覧ビューではこれをエスケープせずにレンダリングします（Str::markdown が安全な HTML にしてくれています）：

```html
{{-- resources/views/ideas/show.blade.php --}}
<div class="prose prose-invert">
    {!! $idea->formatted_description !!}
</div>
```

注意：エスケープを回避する `{!! !!}` は明確な意図を持って使ってください ― 使うのは信用できる内容か安全な HTML に明示的に変換したもの（Laravel の Markdown ヘルパーはデフォルトで安全な HTML を提供します）に対してだけにしましょう。

## Tailwind のプラグインで文字を美しくする

Markdown の出力を美しく見せるために、Tailwind の typography プラグインを導入しました：

```bash
npm install @tailwindcss/typography
```

これを Tailwind の設定ファイルで登録します：

```javascript
// tailwind.config.js
module.exports = {
  // ...
  plugins: [
    require('@tailwindcss/typography'),
    // ...
  ],
};
```

それから上で示したように、フォーマットされた HTML をラップするコンテナに `prose`（とダークテーマ用に `prose-invert`）クラスを付与します。

## アクセサの動作を確認する簡単なユニットテスト

フォーマットの振る舞いを確認する簡単なテストを用意しました。Pest（または PHPUnit）を使うとこのようになります：

```php
// tests/Unit/IdeaTest.php (Pest スタイル)
it('formats the description using markdown', function () {
    $idea = new App\Models\Idea([
        'description' => "Hello **world**",
    ]);

    expect($idea->formatted_description)->toContain('<strong>');
});
```

PHPUnitでは：

```php
public function test_formats_description_using_markdown()
{
    $idea = new \App\Models\Idea(['description' => "Hello **world**"]);
    $this->assertStringContainsString('<strong>', $idea->formatted_description);
}
```

テストを実行し、調整を加え、コミットしてプッシュすれば、Forge が変更を自動的にデプロイしてくれます。


## 最後のワークフローのまとめ

フォーマッタを実行し、テストを実施し、誤りを修正し、コミットし、プッシュし、Forge のデプロイの様子を見守りましょう。
デプロイに失敗したときは、Forge のログが役立ちます：環境のミスマッチがそこに示されているでしょう（SQLite と MySQLの違い、環境変数の欠如、ビルドエラーなど）。
マークダウン機能ではこれらを追加しました：
- モデルにアクセサを実装し、
- フォーマットされた HTML を Blade でレンダリングし、
- よりよい見た目にするためにオプションで Tailwind の typography を導入し、
- 振る舞いを確認する簡単なユニットテストを用意しました。

以上です ― 小さな機能をアイデアから実現し、テストを行って正しくデプロイまで行いました。
必要であれば、そのまま利用できるパッチとして、私が使ったコード（モデルのアクセサ、Blade のコード、Tailwind の config、そしてテスト）をお渡しできます。
