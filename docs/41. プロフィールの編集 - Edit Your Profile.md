# [プロフィールの編集](https://laracasts.com/series/laravel-from-scratch-2026/episodes/41)

プロフィールの編集について話しましょう ― ナビゲーションバーにリンクを追加し、ルート＋コントローラ＋ビューを紐づけ、バリデーションと保存を行い、最後にメールアドレスが変更されたときに編集前のアドレスに通知を送るようにします。
小さな機能ですが、確認しておくべきちょっとしたポイントがたくさんあります。

## プロフィールの編集にリンクを追加

ユーザがログインしていたら、ナビゲーションバーに自分のプロフィール編集画面のリンクを載せましょう。
私はよく nav コンポーネントでこのようにします（Blade）：

```html
@auth
  <a href="{{ route('profile.edit') }}" class="link">Edit Profile</a>
@endauth
```

あとで URL を変更したときにハードコーディングした文字列を探し回らないですむように、名前付きルートを使いましょう。

## ルートとコントローラ

ルートをいくつか追加しましょう：編集フォームを表示するものと更新処理を行うものです。
認証を必須にするのを忘れないでください。

```php
// routes/web.php

Route::middleware('auth')->group(function () {
    Route::get('/profile/edit', [App\Http\Controllers\ProfileController::class, 'edit'])->name('profile.edit');
    Route::patch('/profile', [App\Http\Controllers\ProfileController::class, 'update'])->name('profile.update');
});
```

シンプルなコントローラを作成します：

```bash
php artisan make:controller ProfileController
```

そしてコントローラの中では：

```php
// app/Http/Controllers/ProfileController.php

use Illuminate\Http\Request;

public function edit()
{
    return view('profile.edit', ['user' => auth()->user()]);
}

public function update(Request $request)
{
    // バリデーションと更新処理をここに（下記参照）
}
```

## 編集画面（フォーム）

編集画面は登録フォームとほぼ同じですが、ログインユーザの情報が事前記入しておきます：

```html
<!-- resources/views/profile/edit.blade.php -->

<form action="{{ route('profile.update') }}" method="POST">
  @csrf
  @method('PATCH')

  <label>Name</label>
  <input name="name" value="{{ old('name', $user->name) }}" />

  <label>Email</label>
  <input name="email" value="{{ old('email', $user->email) }}" />

  <label>New Password (optional)</label>
  <input name="password" type="password" />

  <button type="submit" class="btn btn-primary">Update Account</button>
</form>
```

Tailwind / DaisyUI を使っているのであれば、入力欄／ボタンのクラスをお好きなコンポーネントクラスに置き換えてください。

## バリデーションと保存

プロフィール更新特有のバリデーション内容は、メールアドレスの重複チェックで現在のユーザのアドレスを除いてチェックすることです（そうしないと現在のアドレスをそのまま使うことができません）：

```php
// ProfileController@update の内部

public function update(Request $request)
{
    $user = $request->user();

    $data = $request->validate([
        'name' => ['required', 'string', 'max:255'],
        'email' => ['required', 'email', 'max:255', 'unique:users,email,' . $user->id],
        'password' => ['nullable', 'string', 'min:8', 'confirmed'],
    ]);

    // パスワードが入力されていたときだけセットする
    if (!empty($data['password'])) {
        $data['password'] = bcrypt($data['password']);
    } else {
        unset($data['password']);
    }

    $user->update($data);

    return redirect()->route('profile.edit')->with('status', 'Profile updated.');
}
```

補足：
- ユーザにパスワード変更をさせるとき、私はいつもフォームに `password_confirmation` を用意します。
- ログインしているだけでプロフィールの更新を許可するのか、もう一度パスワードを入力させるのかは各自で決めてください。これはあなたのアプリのセキュリティの問題です。

## 変更されたことを変更前のメールアドレスに通知する

やっておくといいこと：ユーザがメールアドレスを変更したら変更前のアドレスに通知するようにしましょう。
そうするともしその変更が悪意のある誰かに行われたものだった場合に対処することができます。
ユーザのものではないアドレスにメールを送るには、Notification の route で直接 mail を指定してください。

メールアドレスが変更されたときのコントローラのコードです：

```php
use Illuminate\Notifications\Notification;
use Illuminate\Notifications\AnonymousNotifiable;
use App\Notifications\EmailChanged;

$originalEmail = $user->email;

$user->update($data);

if ($originalEmail !== $user->email) {
    // もとの（持ち主不明の）アドレスにメールを送る
    Notification::route('mail', $originalEmail)
        ->notify(new EmailChanged($user, $originalEmail));
}
```

`EmailChanged` という通知は、ユーザともとのメールアドレスを受け取って、もしこの変更に身に覚えがなかったときにユーザがサポートに連絡を取る方法を伝える有用なメールを作成することができます。

## プロフィール更新と通知のテスト

テストで本物のメールが送信されないように、Laravel の Notification fake を使って通知が送られたかどうかを確認しましょう。
例（Pest/PHPUnit 形式）：

```php
use Illuminate\Support\Facades\Notification;
use Illuminate\Notifications\AnonymousNotifiable;
use App\Notifications\EmailChanged;

public function test_profile_update_sends_notification_to_original_email()
{
    Notification::fake();

    $user = User::factory()->create([
        'email' => 'old@example.com',
    ]);

    $this->actingAs($user)
         ->patch(route('profile.update'), [
             'name' => 'New Name',
             'email' => 'new@example.com',
         ]);

    // 通知が元のメールアドレスに送られることを想定
    Notification::assertSentTo(
        new AnonymousNotifiable,
        EmailChanged::class,
        function ($notification, $channels, $notifiable) {
            // 通知の宛先が元のメールアドレスになっていることを確認
            return isset($notifiable->routes['mail']) && $notifiable->routes['mail'] === 'old@example.com';
        }
    );
}
```

ユーザのレコードが更新されたことも確認しましょう：

```php
$this->assertEquals('New Name', $user->fresh()->name);
$this->assertEquals('new@example.com', $user->fresh()->email);
```

ブラウザテストでリダイレクトとフラッシュメッセージの挙動をテストするのか、シンプルに手動でリクエストしてチェックするのかはお好みできめてください。

## UX とセキュリティについての補足

大事な情報（メールアドレス／パスワード）を変更するときにパスワード入力を求めるのかどうかを早い段階で決めましょう。
多くの小さなアプリではログインしていることだけで十分です；よりセキュリティが求められるアプリでは再認証が必要です。

メールアドレスを変更したら元のメールアドレスに通知しましょう。
どこまで厳格にしたいかに応じて、セッションをリセットする、新しいアドレスに確認リンクを送る、新しいアドレスの検証を必要とするなどの処理を検討しましょう。

## まとめ

ナビゲーションバーに編集リンクを追加し、ルートとコントローラを作成し、プロフィール編集画面を用意し、変更内容を検証してから安全に保存し、メールアドレスが変更されたときにもとのアドレスに通知できるようにしました（さらにその振る舞いのテストも）。
小さな機能ですが、たくさんの実践的内容がつまっています ― アプリの信頼性を高める素晴らしい内容です。
