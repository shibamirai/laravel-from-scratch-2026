# Pest で登録フォームのブラウザテストを行う

実際に動作するユーザ登録、ログイン、ログアウトの作成について話しましょう ― 不完全なサインアップフローのままリリースしないように認証フォームのブラウザテストを行います。
アカウントを作ってサインインできなければこのアプリは役に立たないので、このフローをエンドツーエンドでテストします。

## 認証のブラウザテスト

> Arrange(準備), Act(実行), Assert(アサート) ― テストの 3A です：テストを準備して、アクションを実行し、結果を確認します。

私はブラウザスタイルのテストを書くのが好みです（Pest には優れたブラウザヘルパーがあります）。
これにより、実際にユーザーが操作するフォームをテストできます：ページにアクセスし、フィールドに入力し、ボタンをクリックし、リダイレクトや認証状態、データベースレコードの状態を確認します。

### セットアップ

Pest のブラウザヘルパーをインストールします（そのパッケージの Pest ドキュメントに従ってください）。
テストが毎回クリーンな状態で実施されるように、必ずテストの開始時処理（Pest config）でデータベースをリフレッシュさせましょう。

tests フォルダ内でブラウザテストのフォルダを分けておくと（tests/Browser など）簡単に見つけて実行できます。

### ユーザ登録のテストを書く

一般的な流れはこのようになります：登録ページにアクセスし、フォームに記入、送信、リダイレクトされることを確認、ログインされていることを確認、DB にユーザが追加されていることを確認。

Pest スタイルでのコードはこのようになります：

```php
// tests/Browser/RegisterTest.php

it('registers a user', function () {
    $this->browse(function ($browser) {
        $browser->visit('/register')
                ->type('name', 'John Doe')
                ->type('email', 'john@example.com')
                ->type('password', 'password123')
                ->type('password_confirmation', 'password123')
                ->press('Create Account') // または特定のボタンを選択
                ->assertPathIs('/')
                ->assertAuthenticated();

        $this->assertDatabaseHas('users', [
            'email' => 'john@example.com',
            'name'  => 'John Doe',
        ]);
    });
});
```

注意点をいくつか：特定のボタンを指定するか、同じボタンが複数個所に現れる場合（ナビゲーションとフォームなど）には data-test 属性を使用します。
そうすることでテストの信頼性を保ちます。

テストの様子が確認できるようにテストで実際にブラウザを開くデバッグヘルパーを使用することもあります ― 最初のテスト開発時に役立ちます：
<p>
  I also sometimes use a debug helper so the test opens a browser while I
  watch — great for initial test development:
</p>

```php
$browser->debug(); // 様子を確認できるよう実際にブラウザを開く
```

### ログインテスト (Arrange, Act, Assert)

データベースにユーザーがいなければログインできません。
テストでは毎回 DB をリフレッシュするので、まずユーザを作成します。

```php
it('logs in a user', function () {
    $user = \App\Models\User::factory()->create([
        'email' => 'jane@example.com',
        'password' => bcrypt('password123'),
    ]);

    $this->browse(function ($browser) use ($user) {
        $browser->visit('/login')
                ->type('email', 'jane@example.com')
                ->type('password', 'password123')
                ->press('@login-button') // その方が便利なら data-test セレクタを使用
                ->assertPathIs('/')
                ->assertAuthenticated();
    });
});
```

もし準備（ユーザの作成）を忘れていれば、DB に一致する認証情報が存在しないためログインテストは失敗します。
テストはそれをすぐに知らせてくれます。

### ログアウトテスト

ログアウトのテストでは、まずユーザを認証し（actingAs を使用するか実際にログインする）、ログアウトボタンをクリックしてユーザがゲストになることを確かめます。

```php
it('logs out a user', function () {
    $user = \App\Models\User::factory()->create();

    $this->actingAs($user)
         ->browse(function ($browser) {
             $browser->visit('/')
                     ->press('@logout-button') // またはセレクタで指定
                     ->assertGuest();
         });
});
```

### 異常系もテストする

正常系のテストは必要ですが、そこで終わってはいけません。
バリデーションや異常系もテストします：メールアドレス未入力、メールアドレスのフォーマット、短いパスワード、認証の誤りなどです。
例えば、メールアドレスのフォーマットが間違っていれば登録ページにとどまったままでフィードバックを表示すべきです。

```php
it('requires a valid email when registering', function () {
    $this->browse(function ($browser) {
        $browser->visit('/register')
                ->type('name', 'John')
                ->type('email', 'not-an-email')
                ->type('password', 'password123')
                ->type('password_confirmation', 'password123')
                ->press('Create Account')
                ->assertPathIs('/register'); // フォームにとどまる
    });
});
```

サーバでバリデーションを行う前に、ブラウザによるクライアント側でのバリデーションによってフォームがブロックされることがあります。
サーバ側でのバリデーションをテストしたければ、テスト中はクライアント側でのバリデーションを停止／バイパスするか、直接サーバエラーを起こします。

## 実践的なコツ

- よくターゲットとされる要素（ボタン、入力項目）には data-test 属性を使いましょう。表示テキストに頼ったセレクタの不安定さがなくなります。
- 前の状態に依存したテストにならないように、必要なデータは準備フェースで毎回作成しましょう。
- 開発中には実施するテストは１つずつにしましょう。エディタ統合機能や、1つのファイル/テストに対して Pest を実行するようなコマンドを使用することで、フィードバックループを大幅に短縮できます。
- ブラウザテストを書いているときは、実際に UI を見ながら確認できるようにデバッグモードを使用しましょう。
- UI の見た目（リダイレクト、表示されるメッセージ）と持続する状態（データベースのレコード、認証状態）の両方を確認しましょう。

## まとめ

ユーザ登録、ログイン、ログアウトのブラウザテストによって認証フローが保証されます。
まずは正常系から始め、バリデーションや認証エラーなどの異常系を追加していきましょう。
テストの焦点を絞り (arrange, act, assert) 、信頼できるセレクタ（data-test 属性）を使用して不確実性をなくしましょう。
