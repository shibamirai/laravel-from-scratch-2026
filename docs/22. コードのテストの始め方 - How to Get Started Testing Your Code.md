# [コードのテストの始め方](https://laracasts.com/series/laravel-from-scratch-2026/episodes/22)

Pest を使った PHP とブラウザでのテストの始め方について話しましょう ― Pest のインストール方法、実際にブラウザを開く機能／ブラウザテストの書き方、それからいつも私が行っているいくつかの実践的なコツ（data-test属性を使ったりデバッグ中にブラウザを一時停止したり）についてです。
テストを始めるためのおすすめの方法です。

## Pest をインストールしてテストを始める

新しく Laravel アプリを作成したら、既に Pest が含まれているはずです。
なければ Pest を取得して初期化します。
プロジェクトルートでこうしてください：

```bash
composer require pestphp/pest --dev
./vendor/bin/pest --init
```

すると Laravel ヘルパーを通じてテストを開始することができます。

```bash
php artisan test
```

もしくは Pest を直接実行します。

```bash
./vendor/bin/pest
```

Pest を初期化すると tests/Pest.php が作られます。
これは テストのために Laravel を起動する TestCase のベースであり、機能テストとユニットテストの例が入っています。
最初は機能テスト（とブラウザテスト）をよく使うことになるでしょう ― ユニットテストはもっと限定的なロジックのテストで、後に役立つでしょう。

## 機能テスト対ユニットテスト

機能テストはアプリを大きな単位で区切って実行します：ページを訪れ、フォームを送信して、何が表示されるか、どこにリダイレクトされるかを確かめます。
ユニットテストはもっと原始的で、コードの小さな単位で行います。
初めてテストに取り掛かるのであれば、最初は機能／ブラウザテストから始めましょう ― これまで手動でやっていたブラウザ上でのテストに近いです。

## ブラウザテスト (Playwright ラッパー)

Pest には Playwright をラップするプラグインがあり、実際にブラウザを開いて操作し、UI がどのようになるかをテストすることができます。
プラグインをインストールし（ライブラリ名や正確なコンポーザーパッケージ名はプラグインによって変わるので、プラグインのドキュメントに従ってください）、そのインストールコマンドを実行して Playwright ブラウザをダウンロードします。

インストールできたらブラウザテストの出力フォルダを .gitignore に追加し、スクリーンショットや動画がコミットされないようにします。

```text
# .gitignore
/tests/Browser
```

ブラウザテストの結果を保持するため、その tests/Browser ディレクトリを作成します。

## 簡単なブラウザテストを書く

私がよく使うパターンは：ページを訪れ、名前か data-test 属性で入力を操作し、送信して、結果を確かめます。

ブラウザテストの例（Pest + ブラウザプラグイン）：

```php
<?php
use App\Models\User;
use function Pest\Laravel\actingAs;

it('registers a user', function () {
    // ブラウザで登録ページを開く
    $this->browse(function ($browser) {
        $browser->visit('/register')
                ->type('name', 'Jane Doe')
                ->type('email', 'jane@example.com')
                ->type('password', 'password')
                ->click('@register-button') // 下記の data-test の使い方を参照
                ->assertPathIs('/ideas');
    });

    // DB にユーザが1件できていることを想定
    expect(User::count())->toBe(1);

    // また、ログインしていることを想定
    $this->assertAuthenticated();
});
```

追記：

- `visit` を呼ぶとページを開き；`type` で入力（名前やセレクターで選択）します；`click('@register-button')` では data-test ショートカット（下記参照）を使用しています
- 機能テストでは 200 応答を確認することが多いですが、ブラウザテストでは表示される HTML やクライアント側での振る舞いを見ます

## data-test 属性 (と `@` ショートカット)の使用

ページ内に同じようなボタンが複数ある場合、テストではそれらを識別できなければいけません。
そこでテスト用に意図的な属性を付与します。

```html
<button data-test="register-button">Register</button>
```

Pest では `[data-test="register-button"]` と書く代わりに `@` をつけてそこにアクセスできます。

```php
$browser->click('@register-button');
```

こうすることでテストが読みやすく明瞭になります：機能の担当者はこの属性を見るとテストに使用されていることを認識します。

## デバッグとブラウザの停止

ブラウザテストを作成しているとき、いつも行うことが２つあります：

まず、実行状況を監視するためにヘッド付きモードでテストを実行します（Pest/Playwright プラグインではデバッグフラグまたはオプションを指定できます）。

次に、ブレークポイント／ポーズメソッドをテストに追加し、ブラウザを開いて操作可能状態にとどめておくようにします。
多くのブラウザプラグインでは `pause()` や `debug()` といったメソッドが提供されており、テストコードで利用できます。
テスト作成中にこのようにインラインで記述します：


```php
$browser->visit('/register')->pause(); // ブラウザを停止して操作可能
```

テストを書き終えたら、ポーズを削除してヘッドレスモードで実行します。

## スクリーンショットとテストの失敗

ブラウザテストで失敗すると、プラグインはスクリーンショットやトレースを取ることが多いです。
.gitignore にスクリーンショット／テスト用フォルダを追加して、これらを Git の対象外にしてください（以前の例を参照）。

## 例：登録機能のテストの手順

まずやることを箇条書きにし、それからそれらをコードに置き換えていきます。

概要を記述：

- 登録ページを訪れる
- 名前、メール、パスワードを入力
- 登録ボタンをクリック
- /ideas にリダイレクトされることを確認
- ユーザが作成されることを確認
- ログインされていることを確認

各行を、対応するブラウザアクションとアサート（以前のサンプルコードを参照）に置き換えます。
これでテストを実行すると、最初は問題が発生するでしょう ― 例えば、ヘッダーには登録リンクがあるので間違った登録ボタンをクリックします。
これこそ `data-test` 属性が価値ある理由です：これによりターゲットが明確になります。

## 一覧と個別表示ページのテスト（アイデアのテスト）

アイデア一覧のテストの例：

```php
it('shows all ideas', function () {
    $user = User::factory()->create();

    actingAs($user);
    $user->ideas()->create([
        'description' => 'Build a thing',
        // 'state' => 'pending' // 明示的にセットしてもモデルのデフォルト値に任せても構わない
    ]);

    $this->get('/ideas')->assertSee('Build a thing');
});
```

`Idea` モデルでデフォルト値をセットしていれば（例えば `state` => `pending` など）、テストレコード作成時に state を省略しても期待通りのテストとなるでしょう。

## デフォルト、Enum、ガード

いくつかのコツ：

カラムにデフォルト値（`state` => `pending` など）が必要なら、モデルでデフォルト値をセットします。

```php
// app/Models/Idea.php
class Idea extends Model
{
    protected $attributes = [
        'state' => 'pending',
    ];

    protected $guarded = []; // オプション：マスアサインメントのガードを外します（要注意）
}
```

後で、より厳密な型チェックと安全性を確保するため、アイデアの状態（保留、実行中、完了）を enum 型にしてコードで参照することを考えます。

## 最後のアドバイスとチェックリスト

ブラウザテストを作成するときは、以下を忘れないでください：

- 明確なセレクタ（`data-test` 属性）で対象を特定する
- フローを文章で書いてから各ステップをテストコードに置き換える
- テストの作成中はブラウザの pause/debug を使用する
- コミット前に debug/pause の呼び出しを削除する
- 正常系だけでなく異常系（バリデーション、メールアドレスの重複、未認証の操作）のテストも追加する

必要であれば、最小限のブラウザテストのファイルと機能テストのサンプル（加えて data-test 属性の追加など Blade テンプレートに加えたちょっとした変更）を提供できます。
