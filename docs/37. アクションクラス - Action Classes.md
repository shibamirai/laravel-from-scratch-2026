# [アクションクラス](https://laracasts.com/series/laravel-from-scratch-2026/episodes/37)

アクションクラスについて話しましょう ― コントローラのアクションを再利用可能にし、テストをしやすくし、役割を分かりやすくするリファクタリングの方法です。
専用のクラスに作成ロジックを取り出し、それをコントローラから利用する方法、それからコンストラクタインジェクションによって現在のログインユーザを受け取るテクニックをお見せします。

## アクションクラスとは何か

> アクションクラスは、ある特定の処理（例えば"アイデアの作成"など）に特化した小さなクラスです。

これはただのクラスです ― 秘密めいたものは何もありません ― ですがクラスに分かりやすい名前 (CreateIdea) をつけることで意図が明確になり、コントローラやコンソールコマンドや AI エージェントから、コードを繰り返し書かなくてもロジックを再利用できるようになります。

## 作成ロジックをクラスに移動する

考え方はこのようになります：アイデアの作成を行うコントローラの store メソッドからすべてを取り出し、それをメソッドを１つだけ持つクラス（私は handle というメソッド名にします）に移動します。

変更前のコントローラ（ロジックを中に直接記述）：

```php
// app/Http/Controllers/IdeaController.php
public function store(IdeaFormRequest $request)
{
    $data = $request->validated();
    $data['image_path'] = $request->file('image')?->store('images');
    $idea = auth()->user()->ideas()->create($data);

    // 手順の作成など...

    return redirect()->route('ideas.index');
}
```

アクションクラスに取り出す：

```php
// app/Actions/CreateIdea.php
namespace App\Actions;

use Illuminate\Support\Facades\DB;
use App\Models\User;
use App\Models\Idea;

class CreateIdea
{
    public function handle(array $attributes, ?User $user = null): Idea
    {
        $user ??= auth()->user();

        // 属性の用意（アップロード画像、デフォルトの手順など）
        if (! empty($attributes['image'] ?? null)) {
            $attributes['image_path'] = $attributes['image']->store('images');
        }

        return DB::transaction(function () use ($user, $attributes) {
            $idea = $user->ideas()->create([
                'title' => $attributes['title'],
                'description' => $attributes['description'],
                'status' => $attributes['status'] ?? 'pending',
                'image_path' => $attributes['image_path'] ?? null,
            ]);

            foreach ($attributes['steps'] ?? [] as $step) {
                $idea->steps()->create(['description' => $step]);
            }

            return $idea;
        });
    }
}
```

こうすることでコントローラがコンパクトになり、いろんなところで作成ロジックを使えるようにします。

## コントローラ：コンテナを使って実行するアクションを解決する

アクションを new してインラインでロジックを呼ぶかわりに、コンテナを使って実行するアクションを解決し、その handle を呼びましょう：

```php
// app/Http/Controllers/IdeaController.php
use App\Actions\CreateIdea;
use App\Http\Requests\IdeaFormRequest;

public function store(IdeaFormRequest $request)
{
    $attributes = $request->only(['title', 'description', 'status', 'image', 'steps']);

    app(CreateIdea::class)->handle($attributes);

    return redirect()->route('ideas.index');
}
```

`app(Class::class)` を使うとインスタンス化処理がコンテナに委ねられ、インスタンス化時の依存関係を考えなくてよくなるため、コードが柔軟になります。

## コンストラクタインジェクションと現在のユーザを表す属性

アクションでは常にユーザが必要になるのであれば、コンストラクタで `User` を受け取ることができます。
Laravel では PHP 属性 に現在の認証ユーザを自動注入することができます。

コンストラクタで注入する例（概念イメージ）：

```php
// app/Actions/CreateIdea.php
use App\Models\User;

public function __construct(
    // Laravel はここに現在のユーザを注入します
    #[\Illuminate\Auth\Attributes\CurrentUser] protected User $user
) {
}

public function handle(array $attributes): Idea
{
    // $this->user はデフォルトで現在のユーザを指すようになっています。
}
```

この方法を使用すれば、別のところでもこのアクションをインスタンス化し、必要に応じて別のユーザを渡すことも可能です（コンテナがデフォルトを解決します）。

補足：上記の attribute は１つのパターンにすぎません ― Laravel のバージョンによっては Attributes クラスが使えますし、明示的に `handle()` にユーザを渡すこともできます。

## アクションクラスをリクエストから切り離す

重要なポイント：アクションクラスは素のデータ（配列や DTO）を受け取るようにしてください。
直接 `Request` を扱ってはいけません。
そうすることで HTTP 以外の、例えばジョブやコマンドやテストなどで再利用することができます。

ですのでフォームリクエストのバリデーションを行った後、バリデーション済みのデータをアクションに渡します：

```php
$attributes = $request->validated(); // または ->only(...)
app(CreateIdea::class)->handle($attributes);
```

## トランザクション保護とちょっとした改善

アクションではしばしば複数の DB 操作を行う（アイデアを作成し、画像を保存し、手順を作成する）ので、それらをトランザクションでラップし、エラー発生時に部分的に更新されてしまうことがないようにしましょう：

```php
return DB::transaction(function () use ($user, $attributes) {
    // アイデアや手順の作成など...
});
```

読みやすくなるように、適宜、途中経過を変数に取り出しましょう；何が起きたか分からない魔法より分かりやすさの方が好まれます。

## テストを用意してリファクタリングを安全に

振る舞いは変わっていないので、既存（アイデア作成用）の機能／ブラウザテストがまだ使えます。
リファクタリング中に何かを壊していないかを確かめるためにテスト一式を使いましょう：

```bash
vendor/bin/pest tests/Browser/CreateIdeaTest.php
```

テストに失敗すれば、失敗した箇所がリファクタリング中に壊してしまったところです ― こうすることで確実にリファクタリングを進めることができます。

## このパターンの利点

アクションクラスによって、コントローラは小さくなり、意図が明確になり、テストしやすく、アプリの別の箇所（コンソールコマンド、キューのジョブ、AI のツールなど）で再利用することができるようになります。
これらは振る舞いの小さな塊で、コードを読みやすく分かりやすくしてくれます。
