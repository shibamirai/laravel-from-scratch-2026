# Eloquent リレーション

アイデアとユーザーのモデリングとリレーションについて話しましょう ― Eloquent でどのように表現するのか、コード（と Tinker）からそれらにどうやってアクセスするのか、それからエディターやコードをもっと分かりやすくするためのいくつかのコツについてです。

## モデルのリレーション

> １個のアイデアは１人のユーザに紐づく
> 各ユーザーはそれぞれ複数のアイデアを持つことができる

この２点がこのドメインの本質です。
プログラム上では、各モデルで Eloquent リレーションメソッドを用いてこれらを表現します。

### Belongs To - 1対1 (Idea → User)

アイデアモデルでは、belongsTo というリレーションによりアイデアに対して誰が作成したのかを問い合わせることができます。

このように定義します：

```php
// app/Models/Idea.php

use Illuminate\Database\Eloquent\Relations\BelongsTo;

public function user(): BelongsTo
{
    return $this->belongsTo(\App\Models\User::class);
}
```

すると、コード内（または Tinker）で取得したアイデアから作者をたどることができます。

```bash
php artisan tinker
>>> $idea = App\Models\Idea::first();
>>> $idea->user; // このアイデアを作成したユーザーを返す
```

微妙な API の違いに注意：user() というメソッドを定義して BelongsTo リレーションを返すようにしていますが、**プロパティ** ($idea->user) としてアクセスすることでその関連モデルのインスタンスを取得しています。 
メソッドとしてアクセス ($idea->user()) とすれば定義どおりリレーションオブジェクト自体を取得します ― これによりクエリーをつなげることができます。

### Has Many - 1対多 (User → Ideas)

ユーザーモデルでは、hasMany リレーションを定義することでそのユーザーが作成したすべてのアイデアが取得できます。


```php
// app/Models/User.php

use Illuminate\Database\Eloquent\Relations\HasMany;

public function ideas(): HasMany
{
    return $this->hasMany(\App\Models\Idea::class);
}
```

Tinker で確認すると：

```bash
php artisan tinker
>>> $user = App\Models\User::first();
>>> $user->ideas; // アイデアのコレクションを返す
>>> $user->ideas[0]->description; // 最初のアイデアの詳細(description)にアクセスする
```

コレクションは "many" リレーションとして返されます―配列としても扱えますし、コレクションヘルパーも使えます。

## リレーションを通じた関連モデルの作成

リレーションを通じて関連モデルを作成しましょう ― Eloquent が自動で外部キーを設定してくれます。

```php
// ログインユーザーが新しいアイデアを作成する
auth()->user()->ideas()->create([
    'title' => 'New idea',
    'description' => 'Quick notes about it',
    'links' => [], // 配列型にしておく
]);
```

こうするとコードとして読みやすいですし手動で user_id をセットする必要がありません。

## 便利なショートカットとヘルパー

Eloquent には確認作業を分かりやすくするヘルパーがあります。
例えば、ID 同士を比較するよりも：

```php
// あまりいい方法ではない
return $user->id === $idea->user_id;
```

モデルが持つ `is()` ヘルパーを使うことができます。

```php
return $user->is($idea->user); // 同一のモデル／テーブルでかつ ID が等しいかどうか
```

こちらの方が分かりやすく、意図を明確に表現しています。

## 戻り値の型 (オプション)

戻り値の型はつけなくても構いませんが、コードの静的分析やエディターの自動補完に役立ちます。

```php
public function user(): BelongsTo
{
    return $this->belongsTo(User::class);
}

public function ideas(): HasMany
{
    return $this->hasMany(Idea::class);
}
```

戻り値の型をつける場合は、ファイルの先頭で関連するクラスをインポートしてください。

## よりよい自動補完のためのエディターのコツ

IDE が動的リレーションシッププロパティへのアクセスを警告する場合（波線や「プロパティが見つかりません」といったエラーが表示される場合）、以下の対処法があります：

一つの方法として、IDE/プラグインに補助メタデータを生成させる方法があります。
PhpStorm 用の Laravel プラグイン（Laravel Idea、Laravel IDE Helperなど）の多くは、動的プロパティに関する情報をエディタに教えるスタブファイルを生成できます。

別の方法としては、PHPDoc で明示的に動的リレーションシッププロパティを記述します。

```php
/**
 * @property \Illuminate\Database\Eloquent\Collection<int, \App\Models\Idea> $ideas
 */
class User extends Authenticatable
{
    // ...
}
```

どちらの方法でも警告はなくなり、自動補完が改善されます。

## なぜこれが大事か（簡単なまとめ）

`belongsTo` や `hasMany` を定義するとモデル間の循環参照を閉じ、コードが分かりやすくなります：アイデアを取得しその作者を尋ねる、ユーザーを取得しその人の作ったものを尋ねる。
リレーションを通じて関連レコードを作成すると (`$user->ideas()->create([...])`) 外部キーが必ず設定され、コードが読みやすくなります。
