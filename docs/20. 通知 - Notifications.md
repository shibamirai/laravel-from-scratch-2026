# [通知](https://laracasts.com/series/laravel-from-scratch-2026/episodes/20)

通知について話しましょう ― アイデアが作成されたときにどうやって通知するか、どうやってデータベースに保存するか、それからローカルでメールを参照する方法についてです。
私が実施した細かな変更点を順を追って説明し、実際にコピー＆ペーストできる具体的なコードスニペットを提供します。

## 通知とは

> 通知により、アプリで発生した何かをユーザに知らせることができます。

通知とは、アラートを送る正式な方法だと考えてください：メール、SMS、またはアプリの通知ウィジェットを動作させるデータベースのレコードです。
ここではシンプルにしましょう：アイデアを作成したときにメールを送信し、データベースに通知を保存する方法をお見せします。

## データベース通知を可能にする

まず、通知トレイトはユーザーモデルですでに利用できるようになっています（notifications(), unreadNotifications(), notify() などのメソッドが使えます）。
ですが通知のマイグレーションを実行する前に `user->notifications()` を呼び出すと、テーブルが存在しないためエラーが発生します。
マイグレーションを作成して実行しましょう。

通知テーブルのマイグレーションを作成し、実行します：

```bash
php artisan notifications:table
php artisan migrate
```

そうすると Tinker でユーザーの通知を確認できるようになります。

```bash
php artisan tinker
>>> $user = App\Models\User::first()
>>> $user->notifications // コレクション (なければ empty が返る)
```

## 通知クラスの作成

通知がどんな内容でどのように運ばれるかを定義する通知クラスを作成します。

```bash
php artisan make:notification IdeaPublished
```

app/Notifications/IdeaPublished.php を開いて通知を作成します。
これはシンプルなメールだけの例で、コンストラクタでアイデアを受け取り、リンクを含んだ MailMessage を作成します。

```php
<?php

namespace App\Notifications;

use Illuminate\Notifications\Notification;
use Illuminate\Notifications\Messages\MailMessage;
use App\Models\Idea;

class IdeaPublished extends Notification
{
    public function __construct(protected Idea $idea)
    {
        //
    }

    public function via($notifiable)
    {
        return ['mail']; // ['database','mail'] などにすることができます。
    }

    public function toMail($notifiable)
    {
        return (new MailMessage)
            ->greeting('Hello,')
            ->line('You published a new idea: ' . $this->idea->title)
            ->action('Read Idea', url('/ideas/' . $this->idea->id))
            ->line('Thanks for using the app!');
    }

    // 'database' チャンネルを使う場合は toArray() を実装します。
    public function toArray($notifiable)
    {
        return [
            'idea_id' => $this->idea->id,
            'title' => $this->idea->title,
        ];
    }
}
```

通知をキューに入れる（メールはバックグラウンドで送信されるようになる）場合は、クラスで ShouldQueue を実装し、キューワーカを設定しましょう。

## アイデアが作成されたときに通知を送る

store アクションでアイデアが作成した後、オーナーに通知します。
例 IdeaController@store（省略しています）：

```php
<?php
use App\Models\Idea;
use App\Notifications\IdeaPublished;

public function store(Request $request)
{
    $attributes = $request->validate([
        'title' => 'required|string|max:255',
        'description' => 'nullable|string',
        'state' => 'nullable|string',
    ]);

    $idea = $request->user()->ideas()->create($attributes);

    // オーナーにアイデアが追加されたことを通知
    $request->user()->notify(new IdeaPublished($idea));

    return redirect()->route('ideas.index')->with('success', 'Idea created.');
}
```

テストのために即座に通知されるようにしたい場合は notifyNow() を使用しますが、本番環境では通知はキューイングした方がいいでしょう。

## ローカルでメールを閲覧し確認する

デフォルトではローカル環境において Laravel はメールをログに書きこみます。
メーラのような UI で実際の HTML メールを参照するには、Mailpit や MailHog、Mailtrap といったローカルで動作するメールキャプチャツールを使用します。
例えば Mailpit の場合（Mac OS で Homebrew を使ってインストール）：

```bash
brew install mailpit
mailpit        # メールの UI を開始
```

それから .env のメール設定で Mailpit の SMTP サーバ（通常は host 127.0.0.1, port 1025）を指定します。

```bash
MAIL_MAILER=smtp
MAIL_HOST=127.0.0.1
MAIL_PORT=1025
MAIL_FROM_ADDRESS="admin@example.test"
MAIL_FROM_NAME="Example App"
```

これで、メールチャンネルからメールが送られると、Mailpit の ウェブ UI 上に表示されます。

ログで確認したい場合は storage/logs/laravel.log を確認します ― 設定に従って Laravel はメールをログに記録します。

## データベース通知を利用する

via() で ['database'] を返した場合、通知は通知テーブルに保存されます。
それらはユーザーモデルからリストアップしたり、開封済みとしてマークすることができます：

```php
$user = User::find(1);

// 未開封の通知
$unread = $user->unreadNotifications;

// 最初の通知を開封済みにする
$user->unreadNotifications->first()->markAsRead();

// すべての通知
$all = $user->notifications;
```

これによりアプリの通知ドロップダウンを動作させます。

## ドライバーと配信オプション

通知はメールとデータベースだけではありません。
Laravel は複数の配信チャネルをサポートしています（メール、データベース、Nexmo/Twilio パッケージを介したSMS、Slack など）。
また、エコシステムには多数のコミュニティ製ドライバーが用意されています。
ニーズやユーザーの好みに応じて、適切なチャネルを選択してください。

メールの送信はバックグラウンドで行いたいという場合は、通知に ShouldQueue インターフェースを実装してください。

```php
use Illuminate\Contracts\Queue\ShouldQueue;

class IdeaPublished extends Notification implements ShouldQueue
{
    // ...
}
```

そしてバックグラウンド処理が行われるようにキューワーカ (redis/beanstalk/whatever) を実行します。

## Tinker で試す

Tinker を使って通知を試すには：

```bash
php artisan tinker
>>> $user = App\Models\User::first();
>>> $idea = App\Models\Idea::latest()->first();
>>> $user->notify(new App\Notifications\IdeaPublished($idea));
```

Mailpit を実行していれば、UI にメールが表示されるでしょう。
データベースチャンネルを使っている場合は、$user->notifications に新しいレコードが追加されているはずです。

## ちょっとしたコツと注意

- ユーザーモデルにはデフォルトで Notifiable トレイトが含まれており、notify(), notifications(), unreadNotifications(), などが使えるようになっています。
- アプリの通知リストを使いたければデータベースの通知テーブルを使用します。メールを使いたければ mail を使用します。非同期通知がしたければ ShouldQueue を使います。
- ローカル環境でメール送信を確認するときは Mailpit や MailHog といったメールキャプチャーツールが便利です ― 外部にメールを送信することなく HTML 表示やリンクのテストが行えます。
- 本番環境ではメール送信のキューイングやリトライを考えましょう；ローカル環境では同期送信でも構いません。

必要であれば、コピーペーストして使えるように、完全な通知クラスのファイル、IdeaController の store の変更点、最低限のメール設定をした .env ファイルの例をお渡しすることができます。
