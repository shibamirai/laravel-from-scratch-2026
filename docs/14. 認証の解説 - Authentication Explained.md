# [認証の解説](https://laracasts.com/series/laravel-from-scratch-2026/episodes/14)

認証について話しましょう — シンプルな登録/ログインフローの骨組みを構築し、ルートとコントローラーを接続し、ビューを紐付け、バリデーション、ハッシュ化、
セッション処理といった面倒な作業を Laravel に任せます。
実際にエディターで行う手順を解説するので、まるで肩越しに見ているかのように感じられるでしょう。

## ルーティングとコントローラー

私は通常、routes ファイルから始めて必要なページを監視します。
登録フローの場合、通常は2つのルートを追加します：1つは登録フォームを表示するため、もう1つはユーザーを保存するためです。
名前は自由に付けられますが、私は RESTful 的な規約に従うようにしています。

以下がその様子です：

```php
use App\Http\Controllers\Auth\RegisteredUserController;

Route::get('/register', [RegisteredUserController::class, 'create']);
Route::post('/register', [RegisteredUserController::class, 'store']);
```

コントローラーを生成するために、以下を実行します：

```bash
php artisan make:controller Auth/RegisteredUserController
```

これにより、app/Http/Controllers/Auth/RegisteredUserController.php が生成されます。
通常、create と store のアクションはそこに配置します。

## 登録フォームの作成

DaisyUI からフィールドセットコンポーネントを取得し、フォームを素早く骨組み化するために resources/views/auth/register.blade.php に貼り付けました。
ナビゲーションでは登録リンクを公開し、登録ルートを指すように設定しました。
またナビゲーションコンポーネントで視覚的な微調整（背景色の変更、影の削除など）を行いました。

フォーム自体は /register への POST 送信を行い、CSRF 対策が必要です。
以下は私が起点として使用したシンプルな Blade フォームです：

```html
<form action="/register" method="POST" class="max-w-md mx-auto">
  @csrf

  <label>
    Name
    <input name="name" type="text" />
  </label>

  <label>
    Email
    <input name="email" type="email" />
  </label>

  <label>
    Password
    <input name="password" type="password" />
  </label>

  <button type="submit">Register</button>
</form>
```

名前、メールアドレス、パスワードを追加したのは、デフォルトのユーザーテーブル移行がそれらを想定しているためです。
もちろんこれらのフィールドは変更可能です — 決めるのはあなたです。

## バリデーションとユーザの作成

まず RegisteredUserController::store の処理手順を概説します：検証し、ユーザーを作成、ログインして、そしてリダイレクトします。

インライン検証を行うには、リクエストを受け取り validate を呼び出します。
いつも私は妥当と思われる制限（例：名前/メールアドレスには VARCHAR(255)）を設定し、メールアドレスの一意性要求を必ずセットします。
パスワードには、単純な min:8 を使用するか、デフォルトを統合した Password ルールを使用します。

以下に整理した store メソッドを示します：

```php
<?php

namespace App\Http\Controllers\Auth;

use App\Http\Controllers\Controller;
use App\Models\User;
use Illuminate\Http\Request;
use Illuminate\Support\Facades\Hash;
use Illuminate\Support\Facades\Auth;
use Illuminate\Validation\Rules\Password;

class RegisteredUserController extends Controller
{
    public function create()
    {
        return view('auth.register');
    }

    public function store(Request $request)
    {
        $attributes = $request->validate([
            'name' => ['required', 'string', 'max:255'],
            'email' => ['required', 'string', 'email', 'max:255', 'unique:users,email'],
            'password' => ['required', Password::defaults()],
        ]);

        $user = User::create([
            'name' => $attributes['name'],
            'email' => $attributes['email'],
            'password' => Hash::make($attributes['password']),
        ]);

        Auth::login($user);

        return redirect('/ideas');
    }
}
```

いくつかの注意点：Hash::make は平文パスワードを絶対に保存しないことを保証します。
validate 呼び出しは検証が成功した場合に検証済み属性を返します。
私はこれを頻繁に再利用しています。

## セッションとログアウト

ログインとログアウトのために SessionsController を作成しました。
リソースコントローラーとして生成し（その後未使用メソッドを削除）、名前は任意で構いません。

```bash
php artisan make:controller Auth/SessionsController --resource
```

```php
use App\Http\Controllers\Auth\SessionsController;

Route::get('/login', [SessionsController::class, 'create']);
Route::post('/login', [SessionsController::class, 'store']);
Route::delete('/logout', [SessionsController::class, 'destroy']);
```

ログイン画面は基本的に登録フォームと同じですが、メールアドレスとパスワードのみを要求します。
store メソッドは認証を試み、成功した場合はリダイレクトし、失敗した場合はエラーを返します：

```php
<?php

namespace App\Http\Controllers\Auth;

use App\Http\Controllers\Controller;
use Illuminate\Http\Request;
use Illuminate\Support\Facades\Auth;
use Illuminate\Validation\Rules\Password;

class SessionsController extends Controller
{
    public function create()
    {
        return view('auth.login');
    }

    public function store(Request $request)
    {
        $credentials = $request->validate([
            'email' => ['required', 'email'],
            'password' => ['required', Password::defaults()],
        ]);

        if (Auth::attempt($credentials)) {
            $request->session()->regenerate();
            return redirect()->intended('/ideas');
        }

        return back()->withErrors([
            'email' => 'The provided credentials do not match our records.',
        ])->onlyInput('email');
    }

    public function destroy(Request $request)
    {
        Auth::logout();

        $request->session()->invalidate();
        $request->session()->regenerateToken();

        return redirect('/ideas');
    }
}
```

いくつか注意点があります：attempt メソッドはブール値を返します。
成功時にはセッションを再生成し、セッション固定攻撃やハイジャックのリスクを低減します。
失敗時には withErrors を使用し、メールアドレスの有無を漏らさずにフォームに分かりやすいエラーを表示できるようにしています。

> intended リダイレクトは、ユーザーをログインページに送られる前に最初にアクセスしようとしたページへ転送します。

## Blade: 認証状態に基づくリンクの表示

既にログインしているユーザーに登録ボタンを表示したくないですよね？
Blade にはそのためのディレクティブが用意されています：@guestと@auth です。
@guest ... @else ... @endguest でロジックを統合できます。

ナビゲーションコンポーネントの例：

```html
<nav>
  @guest
    <a href="/register" class="btn btn-primary">Register</a>
    <a href="/login" class="btn btn-secondary">Log In</a>
  @else
    <form action="/logout" method="POST">
      @csrf
      @method('DELETE')
      <button type="submit" class="btn btn-ghost">Log Out</button>
    </form>
  @endguest
</nav>
```

これによりUIの一貫性が保たれます：ゲストには登録/ログインが表示され、認証済みユーザーにはログアウトが表示されます。

## フォームでのバリデーションエラーの表示

ユーザーにバリデーションエラーを表示する必要があります。
私はエラーコンポーネントを使用しましたが、考え方は単純です：エラーが発生したフィールド（例：メールアドレス）のエラーを表示します。

Blade では次のようにできます：

```html
<x-input name="email" />
<x-errors name="email" />
```

あるいは定番の：

```html
@if ($errors->has('email'))
  <div class="error">{{ $errors->first('email') }}</div>
@endif
```

認証に失敗した場合、そのメッセージをメールフィールドに添付し、ユーザーが即座にフィードバックを確認できるようにします。

## 簡単なまとめ

基本的なフロー全体をカバーしました：ルートとコントローラーのスケルトン生成、シンプルな登録・ログインフォームの作成（DaisyUI でマークアップのスケルトン生成を支援）、リクエストの検証、パスワードのハッシュ化、Eloquent によるユーザー作成、Auth::login または Auth::attempt によるログイン処理、セッションの再生成、@guest/@auth によるナビゲーションリンクの表示/非表示処理です。

次回の内容：アカウントを作成したので、すべてのアイデアを表示するのではなく、"自分のアイデア"(認証されたユーザのアイデア)だけを表示すべきです。
