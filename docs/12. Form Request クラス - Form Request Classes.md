# [Form Request クラス](https://laracasts.com/series/laravel-from-scratch-2026/episodes/12)

フォームリクエストクラスについて話しましょう — コントローラーから検証を抜き出して専用クラスに移すタイミング、その仕組み、そして便利なヒントをいくつか紹介します。
最初に一言：フォームリクエストクラスを使うことが、あなたをより優れた開発者にするわけではありません。
単なる選択肢です。
プロジェクトやチームに合ったアプローチを選んでください。

> フォームリクエストは、特定のフォームに対する認証ロジックと検証ルールをカプセル化したカスタム Request クラスであり、コントローラー内で型指定されると自動的に検証を実行します。

## フォームリクエストの使用タイミング

コントローラー内でインライン検証を行うか、専用のフォームリクエストで検証できます。
道徳的にどちらの方法が優れているということはありません。
コードをより明確にする方を選択してください。
私は通常、検証が複雑な場合や再利用する場合、あるいは検証と認可を一箇所にまとめたい場合にフォームリクエストを採用します。

## フォームリクエストの作成

Artisan でフォームリクエストを作成できます。
「アイデアを保存する」フローの場合、私は StoreIdeaRequest と名付けました：

```bash
php artisan make:request StoreIdeaRequest
```

これにより、App\Http\Requests\StoreIdeaRequest.php が作成されます。

## フォームリクエストの内容とは

フォームリクエストには主に2つのフックがあります：認証メソッドとルールメソッドです。
デフォルトで生成されるクラスは以下のような構造になります（簡略化しています）：

```php
<?php

namespace App\Http\Requests;

use Illuminate\Foundation\Http\FormRequest;

class StoreIdeaRequest extends FormRequest
{
    public function authorize()
    {
        // リクエストを許可する場合は true を返す。false を返すと 403 となる。
        return true;
    }

    public function rules()
    {
        return [
            'description' => ['required', 'string', 'min:4'],
            'title' => ['required', 'string', 'max:255'],
            // ...その他のルール...
        ];
    }
}
```

authorize() を使用して、ユーザーがこのリクエストを実行する権限があるかどうかを確認します（例：プロジェクトメンバーのみがアイデアを追加できる）。
authorize() が false を返した場合、リクエストは 403 エラーで失敗し、検証は実行されません。

## 検証は自動的に実行される

ここがポイントです：フォームリクエストは、通常コントローラーで型指定するリクエストインスタンスを拡張します。
つまり、通常の Request 型指定をフォームリクエストに置き換えるだけで、validate() を自分で呼び出す必要はありません。

```php
// 変更前
public function store(Request $request)
{
    $validated = $request->validate([/* ルールを書く */]);
    // ...
}

// 変更後
public function store(\App\Http\Requests\StoreIdeaRequest $request)
{
    // バリデーションはすでに実行済み
    $data = $request->validated();
    // ...
}
```

フレームワークは、コントローラーが実行される前にフォームリクエストを検証し、検証に失敗した場合にリダイレクトまたは例外をスローします。

## エラーメッセージと属性のカスタマイズ

カスタムメッセージが必要な場合は、messages() をオーバーライドしてください。

```php
public function messages()
{
    return [
        'description.required' => 'Description is required.',
        'description.min' => 'Give me something at least :min characters long.',
    ];
}
```

属性名は attributes() を通じてより適切な名前に変更することもできます：

```php
public function attributes()
{
    return [
        'description' => 'idea description',
    ];
}
```

それらのプレースホルダートークン（:min など）とドット構文（field.rule）により、柔軟で読みやすいメッセージを実現できます。

## リクエストクラスは１つ？それとも２つ？

作成と更新のルールが異なるかどうかに基づいて決定してください。
ルールが同一の場合、単一のクラスを共有します（クラス名を IdeaRequest に変更）。
ルールが異なる場合（例：更新時に異なる動作をする固有のルール）、StoreIdeaRequest と UpdateIdeaRequest を作成します。

クラスを共有する場合：

```php
// app/Http/Requests/IdeaRequest.php
// これを両方のコントローラメソッドで使用する
public function store(\App\Http\Requests\IdeaRequest $request) { ... }
public function update(\App\Http\Requests\IdeaRequest $request, Idea $idea) { ... }
```

分割した場合は、それぞれが独自の rules() と authorize() ロジックを持ちます。

## いくつかの実用的なポイント

- authorize() 内部で認証チェックを実行できます。これは、検証実行前にエンドポイントを保護するのに便利です。
- authorize() から false を返すと 403 が返され、検証とコントローラーの実行が阻止されます。
- rules() メソッドは、既にご存知の検証構文を使用します：required, email, unique:table, column, min: など。
- フォームリクエストは、検証メッセージのオーバーライド、属性名、さらにはカスタム検証準備メソッド (prepareForValidation()) を集中管理するのに適しています。

## 例：カスタムメッセージと認証

以下に、authorize と messages の動作を示す簡単ですが完全な例を示します：

```php
class StoreIdeaRequest extends FormRequest
{
    public function authorize()
    {
        // アイデアの作成は認証ユーザのみ可能とする
        return auth()->check();
    }

    public function rules()
    {
        return [
            'title' => 'required|string|max:255',
            'description' => 'required|string|min:4',
        ];
    }

    public function messages()
    {
        return [
            'description.required' => 'Come on, gimme something.',
            'description.min' => 'Give me at least :min characters for the description.',
        ];
    }
}
```

これをコントローラーに組み替えて動作を確認してください：
欠落フィールドはエラーと共にリダイレクトされ、未認証の呼び出しは403を返し、メッセージは設定内容と一致します。

## まとめ

フォームリクエストは、特定のフォームに対する検証と認可をまとめる便利な場所です。
それ自体が優れたものにするわけではありません—単に整理を助けるだけです。
保存と更新で検証が同一なら、単一のリクエストを再利用してください。
ルールが異なる場合は分割しましょう。

必要なら、すぐに使える StoreIdeaRequest と、交換前後の比較を示す小さなコントローラー例を貼り付けられます。
