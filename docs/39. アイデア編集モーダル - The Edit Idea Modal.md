# [アイデア編集モーダル](https://laracasts.com/series/laravel-from-scratch-2026/episodes/39)

アイデアの編集について話しましょう ― “アイデア作成” モーダルを編集で再利用します。
気になるのは：同じアクション／モーダルが更新で使えるのか、それとも別のフローが必要なのか？です。
まずはテストから始め、UI をテスト可能にし、それから実際の問題を修正します：閲覧ページにはモーダルがないという問題です。
これはつまりイベントを発生させても何も起こらないということです。
そこでモーダルを再利用可能なコンポーネントに取り出し、必要なところに埋め込みます。

## テストから始める

必要な振る舞いを記述したテストを書くところから始めましょう。
作成時のテストをコピーして、既存のアイデアを編集するときのブラウザ／機能テストを作りましょう。
テストでは、ユーザをログインさせ、そのユーザのアイデアを作成し、そのアイデアの閲覧ページを訪れ、編集をクリックします。

大まかな例がこちらです（Pest/ブラウザスタイルの疑似コード）：

```php
// tests/Browser/EditIdeaTest.php (抜粋)
public function test_can_open_edit_modal_for_existing_idea()
{
    $user = User::factory()->create();
    $idea = Idea::factory()->for($user)->create();

    $this->actingAs($user)
         ->visitRoute('ideas.show', $idea)
         ->click('@edit-idea-button') // 追加予定の data 属性
         ->assertSee('Update Idea')   // モーダルには更新用のラベルが表示される
         ->screenshot('edit-modal-open');
}
```

ボタンが見つからないと言ってテストが失敗すれば、それはつまりテスト対象を特定できるセレクタが UI に必要だということです。
これがテストに必要となる最初のちょっとした修正です。

## 編集ボタンをテスト可能にする

編集ボタンに data 属性をつけて、ブラウザテストで使えるようにします。
同時にボタンから modal-open イベント（アプリでモーダルを開くきっかけ）を出すようにしておきましょう。

簡単なボタンの例：

```html
<!-- resources/views/ideas/show.blade.php -->
<button
  data-edit-idea-button
  x-data
  @click="$dispatch('open-modal', { name: 'create-idea' })"
  type="button"
  class="btn"
>
  Edit Idea
</button>
```

CSS クラスやテキストの内容などの変わりやすい内容に頼らずにテストで参照（->click('@edit-idea-button') できるように、分かりやすい data 属性(data-edit-idea-button) を付与します。

## モーダルの可用性となぜイベントが働かないのか

つまづきやすいポイント：イベントを発生させても、イベントリスナーが現在のページになければ意味がありません。
このアプリではインデックスページにしかモーダルがありませんでした ― 下部にハードコーディングされています。
閲覧ページにはモーダルが配置されていないので、発生した open-modal イベントは行き場を失っていました。
これがテストで（および手動でクリックしたときに）モーダルが開かなかった理由です。

つまり修正すべきは、モーダルを必要なところで使えるようにすることです ― すべてのページにコピーする（やめて！）か、もっといいのは、色んなページで使えるように再利用可能なコンポーネントとして取り出しましょう。

## モーダルをコンポーネントに取り出す

モーダルの記述を Blade コンポーネントに取り出しました (resources/views/components/idea/modal.blade.php)。
そしてこれを、モーダルを開く必要があるページ（インデックス、閲覧など）に追加します。
そうすることで Alpine のリスナーが必要なページに配置されます。

コンポーネントの使い方はこのようになります：

```html
<!-- インデックスページに含める -->
<x-idea.modal name="create-idea" title="New Idea">
  <!-- フォームスロット (登録フォーム) -->
</x-idea.modal>

<!-- 閲覧ページに含める -->
<x-idea.modal name="create-idea" title="Update Idea">
  <!-- フォームスロット (編集のため事前記入されたもの) -->
</x-idea.modal>
```

そしてモーダル自身はウィンドウイベントを待ち受けます：

```html
<!-- resources/views/components/idea/modal.blade.php (抜粋) -->
<div
  x-data="{ show: false, modalName: @js($name) }"
  x-on:open-modal.window="if ($event.detail?.name === modalName) show = true"
  x-show="show"
  style="display:none;"
  role="dialog"
  aria-modal="true"
>
  <!-- モーダルのマークアップとフォームスロット -->
  {{ $slot }}
</div>
```

これで編集ボタンが "create-idea" という名前で open-modal イベントを発生させると、閲覧ページのモーダルがそれを受け取って動作します。

## 編集用にモーダルを事前記入させるコツ

編集用にモーダルを開く時は、既存のアイデアの内容が記入された状態になっていてほしいですよね。
これにはいくつか方法があります：

- モーダルコンポーネントに編集フォームを含め、初期値をサーバからコンポーネントに渡す（Blade の props か JSON で）。
- アイデアの内容を event.detail に持たせたイベントをモーダルで受け取るようにし、表示される前に Alpine を使って記入する。

サーバ側でレンダリングを行うアプリでは、モーダルを閲覧ページに含めるときに事前記入された HTML を渡すのが一番簡単です。

## 再度テストを実施

data 属性が追加され、モーダルが再利用可能なコンポーネントとして取り出され、閲覧ページにモーダルが追加されたので、前回失敗したテストも、今回は編集ボタンが見つかり、イベントが発生し、モーダルが開くことを確認できるはずです。

もしまだテストに失敗するようであれば、テスト中に DOM を調べ（スクリーンショットかデバッグ）、３つのことを確認してください：ボタンのセレクタが存在すること、ページにモーダルのマークアップが存在すること、そしてモーダルの Alpine のリスナーで `name` が一致するウィンドウイベントを待ち受けるようになっていることです。

## まとめ

- 期待する振る舞いを記述したテストを用意するところから始めましょう。
- UI をテスト可能にしましょう（data 属性が便利です）。
- 忘れないでください：イベントは DOM にリスナーが存在するときだけ働きます ― そのページにモーダルがないといけません。
- 必要なところで使えるようにモーダルをコンポーネントに取り出しましょう（インデックス、閲覧など）。
- 編集前に値を事前記入しておくには、Blade の props を使うか、アイデアデータをイベントに含めて Alpine で記入するようにしましょう。

必要であれば、そのままプロジェクトに入れて使えるように、モーダルの Blade コンポーネントのファイルと data 属性に紐づけたテストの例をお渡しできます。
