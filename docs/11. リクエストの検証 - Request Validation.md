## サーバーサイドバリデーション

現在、説明欄に何も入力せずにフォームを送信できてしまいます。
これは問題です。
SQL クエリが実行される段階で、NULL 許可されていない列に NULL が渡されるためデータベースがクラッシュします。
これは**良くありません** — データを永続化する前に、無効な入力をキャッチすべきです。

まずサーバー側で検証します。
グローバルヘルパーを呼び出すか、コントローラーで Request インスタンスを受け取り、その validate を呼び出すかのいずれかです。
どちらも同等です。
以下はコントローラーストアアクション内でインライン化した例です：

```php
<?php
// app/Http/Controllers/IdeaController.php

use Illuminate\Http\Request;

public function store(Request $request)
{
    $attributes = $request->validate([
        'description' => ['required', 'string', 'min:10'],
        // 必要に応じて別のルールを追加
    ]);

    // $attributes は検証済みなので使っても安全です
    $request->user()->ideas()->create($attributes);

    return redirect()->route('ideas.index')->with('success', 'Idea created.');
}
```

上記のルールは、詳細が必須であり、文字列でなければならず、かつ少なくとも10文字以上である必要があることを意味します。
最小値（または最大値）は、アプリに適した値に調整してください。
検証に失敗した場合、Laravel は自動的に前のページへリダイレクトします。
この動作はフォームの一般的なデフォルトです。

> 検証メッセージはセッションにフラッシュ表示され、次のリクエストで使用された後、クリアされます。

これが Laravel が検証メッセージをビューに渡す仕組みです：メッセージは単一のリクエストサイクルの間だけセッションに保存されるため、一度レンダリングすれば消えます。

## なぜ "required" 以上のルールが必要なのか？

required のみのフィールドの場合、ユーザーは「Q」や意味のない文字列を入力してもレコードが作成されてしまいます。
そのため、min:X などの追加ルールやより具体的な制約を設定します。
求めるデータの質を考えてください：文章、段落、それとも単純なタグが必要ですか？
ルールはその意図を反映させるべきです。

## Blade でのバリデーションエラーの表示

エラー表示には複数の選択肢があります。
私は通常、入力フィールドの下にインラインフィードバックを表示することを好みます。
Laravel では全てのビューで $errors オブジェクトが提供されるため、直接参照できます。

以下が長い方法（手動）です：

```html
<!-- resources/views/ideas/create.blade.php -->

<textarea name="description">{{ old('description') }}</textarea>

@if ($errors->has('description'))
  <p class="mt-1 text-sm text-red-500">{{ $errors->first('description') }}</p>
@endif
```

そして、これが Blade ディレクティブを使った簡潔な記述法です。
私はこの簡潔さが気に入っています：

```html
<textarea name="description">{{ old('description') }}</textarea>

@error('description')
  <p class="mt-1 text-sm text-red-500">{{ $message }}</p>
@enderror
```

@error ディレクティブは、そのブロック内で $message 変数を提供します。
これは該当フィールドの最初の検証エラーです。

検証に失敗した際にユーザーが入力したテキストが失われないよう、入力フィールドを再表示するには old('field') を使用することを忘れないでください。

## 再利用可能なエラーコンポーネント

このパターンは多くのフォームで繰り返されるため、小さなコンポーネントとして抽出します。
これによりマークアップの一貫性が保たれ、テンプレートが整理されます。

例えばこのようなコンポーネントを作成します。
resources/views/components/forms/error.blade.php：

```php
@props(['name'])

@error($name)
  <p class="mt-1 text-sm text-red-500">{{ $message }}</p>
@enderror
```

そしてこのように使用します：

```html
<textarea name="description">{{ old('description') }}</textarea>
<x-forms.error name="description" />
```

後でアイコンを追加したり間隔を変更したい場合は、コンポーネント内で一度更新してください。

## フォームの整理のコツ

複数のフォーム関連コンポーネント（textarea、input、select、エラー、ヘルプテキスト）がある場合、それらを components/forms ディレクトリにグループ化し、ドット表記で参照してください。
例：<x-forms.error />。
これにより、発見性と一貫性が保たれます。

## バリデーションルールリファレンス

組み込みルールは数多く存在します — min, max, email, confirmed, in, ファイル/画像ルール, 数値範囲などです。
これらを覚えようとはしないでください。
ドキュメントをブックマークし、特定の制約が必要な時に参照してください。

完全なリストについては公式のバリデーションドキュメントをご覧ください：
[https://laravel.com/docs/validation](https://laravel.com/docs/validation)
