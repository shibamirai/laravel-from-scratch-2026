# フォーム

フォーム処理について話しましょう — これが多くの人のウェブ開発の可能性を広げる鍵です。
新たな章、新たなシャツ、そしてこれから小さなデジタルノートブックを作ります：アイデアを入力し、送信し、どこかに保存する場所です。
この動画こそ、多くの人にとって理解が深まるきっかけになるはずです：
データを受け取り、フォームを送信し、そのデータで何かを行い、完了後にリダイレクトするプロセスを学ぶのです。

## シンプルなフォームの追加

ルートファイルから始めます。ホームページでは「ideas」というビューが読み込まれます。
このレッスンではビューはシンプルです：テキストエリアと送信ボタンだけです。
CSS をたくさん書く様子を見てもらうのは避けたいので、見た目を良くするために Tailwind CSS から少しスタイルを借用します。
Tailwind が気に入らない場合はスキップしてください — フォームのロジックは同じように動作します。

基本的なルートとフォームは次のようになります。

ページを表示するルート:

```php
// routes/web.php
use Illuminate\Support\Facades\Route;

Route::get('/', function () {
    return view('ideas'); // resources/views/ideas.blade.php
});
```

Blade で作成したシンプルな Tailwind スタイルのフォーム（分かりやすくするため簡略化）：

```php
<!-- resources/views/ideas.blade.php -->
<form method="POST" action="/ideas" class="max-w-xl mx-auto p-6 bg-gray-700">
    @csrf

    <label for="idea" class="block text-sm font-medium text-white">
        New Idea
    </label>

    <div class="mt-1">
        <textarea id="idea" name="idea" rows="4"
            class="block w-full rounded-md bg-white text-black"></textarea>
    </div>

    <div class="mt-3">
        <button type="submit" class="inline-flex justify-center px-4 py-2 bg-blue-600 text-white rounded">
            Save
        </button>
    </div>
</form>
```

Tailwind CSS のデモ用スタイルを素早く適用したい場合は、CDN をレイアウトに含めることができます（デモ専用です）：

```html
<!-- <head>タグ内に挿入 (本番環境では使用しない) -->
<link href="https://cdn.jsdelivr.net/npm/tailwindcss@latest/dist/tailwind.min.css" rel="stylesheet">
```

## フォームの送信とルート

デフォルトではフォームは現在のページに対して GET リクエストを送信します。
/ideas への POST リクエストを送信したいので、フォーム送信を受け取る POST ルートを追加してください。

開発中に POST リクエストが正しく送信されているか確認するために私が使用している POST ルートは以下の通りです：

```php
// routes/web.php
use Illuminate\Http\Request;

Route::post('/ideas', function (Request $request) {
    // 開発時の動作チェック:
    dd('hello');

    // 後で $request->input('idea') を実行します
});
```

送信した際に 404 エラーが発生した場合、通常は単にその HTTP メソッド/URI に対するルートをまだ登録していないことを意味します。

## CSRF 対策

Laravel はデフォルトでクロスサイトリクエストフォージェリ攻撃から保護します。
CSRF トークンなしで POST 送信すると、419（ページ期限切れ）レスポンスが返されます。
私も最初は混乱したので、明確に説明します：

> CSRFトークンは、リクエストが現在のサイトから発信されたものであることを検証するために、フォームに追加されるサーバー生成のトークンです。

フォーム内に Blade ディレクティブ @csrf を追加すると、Laravel はそのトークンを含む隠し入力フィールドを生成します。
フォーム送信時、サーバーはフォーム内のトークンと期待するトークンを比較し、一致した場合にのみリクエストを処理します。

@csrf は HTML に以下のようなものを生成します：

```html
<input type="hidden" name="_token" value="SOME_LONG_CSRF_TOKEN">
```

したがって、POST/PUT/PATCH/DELETE フォームには常に @csrf を必ず含めてください:

```html
<form method="POST" action="/ideas">
    @csrf
    <!-- ... -->
</form>
```

それを含めると 419 エラーは解消され、リクエストを処理できるようになります。

## リクエストデータへのアクセス

Laravel では送信されたデータを取得する方法が複数あります。
ヘルパー関数 `request()` は簡潔で便利なので、これらの例ではこれを使用します。

例：リクエストからideaを取得する：

```php
Route::post('/ideas', function () {
    $idea = request('idea'); // request()->input('idea') と同じ
    dd($idea);
});
```

実際のコードでは他のパターンも見られます：Request ファサードの使用、Illuminate\Http\Request をルート/コントローラメソッドのパラメータとしてインポート、またはIlluminate\Support\Facades\Request 経由での Request ファサードの使用です。
いずれも同じ役割を果たします。
好みのスタイルを選んでください。

## セッションにおける持続的なアイデア

データベースについてはまだ説明していませんので、現時点ではアイデアをセッション（ブラウザごとのメモ帳）に保存しています。
一時的な方法ですが、学習には最適です。

以下は、アイデアをセッション配列にプッシュし、ホームページへリダイレクトする POST ハンドラです：

```php
use Illuminate\Http\Request;

Route::post('/ideas', function (Request $request) {
    $idea = $request->input('idea');

    // アイデアを "ideas" という名前でセッション配列にプッシュ
    session()->push('ideas', $idea);

    // ホームページへリダイレクト (フォームとアイデアを表示)
    return redirect('/');
});
```

セッションから読み取り、アイデアをビューに渡す GET ルート（安全なデフォルト値として空の配列を使用）：

```php
Route::get('/', function () {
    $ideas = session('ideas', []); // デフォルトは空の配列
    return view('ideas', ['ideas' => $ideas]);
});
```

Blade ビューで、アイデアのループ処理とレンダリングが可能です：

```html
<!-- resources/views/ideas.blade.php -->
@if (count($ideas) > 0)
    <h3 class="mt-6 font-bold text-white">Your Ideas</h3>

    <ul class="mt-4 space-y-2">
        @foreach ($ideas as $idea)
            <li class="text-sm text-gray-200">{{ $idea }}</li>
        @endforeach
    </ul>
@endif
```

覚えておいてください：セッションは各ブラウザごとに一意です。
別のブラウザやシークレットウィンドウでページを開くと、同じアイデアは表示されません。
長期的な永続化については、後の章でデータベースを使用します。

## アイデアの削除（クイック開発ヘルパー）

開発中は、状態を素早くリセットできるよう、セッションをクリアする簡易ルートをよく追加します。
これは RESTful ではありません（GET リクエストで破壊的動作を実施している）— 学習中の一時的な便宜策に過ぎません。

```php
Route::get('/ideas/delete', function () {
    session()->forget('ideas');
    return redirect('/');
});
```

次のレッスンで戻ってきて、フォームからのメソッド偽装による DELETE リクエストを用いた正しい削除方法を示します。

## 注記と前提

セッションはユーザーごとに固有のメモ帳のようなものだと考えられます。
一時的に何かを書き留めることができます。
データベースは長期的な永続化のためのものです。
ここで不明点があった場合（CSRF、リクエストヘルパー、セッションなど）、ここで質問して理解を深めてから次に進むことをお勧めします。

正確に再現したい場合、Bladeフォームで必要な主要な要素は：メソッドと @csrf ディレクティブ、送信を受け付ける POST ルート、入力値の保存（現時点ではセッションプッシュ）、そしてアイデアを表示するページへのリダイレクトです。

次のレッスンでお会いしましょう。
削除処理を適切に実装し、その後データベースへの永続化に移行します。
