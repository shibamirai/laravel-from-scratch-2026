# 具体的手順

具体的手順について話しましょう ― アイデア実現に向けて行うべきことを記した、チェックリスト形式の小さなアイテムです。
アイデア登録時にモーダル内でこれらの手順を追加し、それをサーバに送信し、アイデアページで表示し、完了したらチェックをつけられるようにしたいと思います。
主にフォームと JavaScript（Alpine.js を使用します）で実装し、コントローラでも少し処理を行います。
コード量は少ないですが、実用的な内容です。

> 手順の内容は、詳細と完了状態です。

これが手順です：内容を記した文章と完了したかどうかの状態です。

## モーダルに手順を追加する

リンク用に作成した HTML を再利用しましょう。
他の入力欄の上に手順のセクションを用意し、新しい手順を追加する入力欄とボタンをモーダルに含めましょう。

新しい手順を Alpine の配列に追加するための入力欄とボタンを記述したフォームの一部がこちらです：

```html
<div class="mt-4">
  <label for="new_step" class="label">Actionable Steps</label>

  <div class="flex gap-2">
    <input
      id="new_step"
      placeholder="What needs to be done?"
      x-model="newStep"
      class="input flex-1"
      autocomplete="off"
      spellcheck="false"
    />

    <button
      type="button"
      class="btn"
      :disabled="!newStep"
      @click="steps.push(newStep); newStep = ''"
    >
      Add
    </button>
  </div>

  <!-- サーバ側でのバリデーションエラーの表示 -->
  @error('steps')
    <p class="text-red-500 text-sm mt-1">{{ $message }}</p>
  @enderror
</div>
```

そしてこのモーダルの Alpine のルートです（モーダルコンポーネントの先頭に配置）：

```html
<div x-data="{ newStep: '', steps: [] }">
  <!-- モーダルの内容 -->
</div>
```

Chrome で Alpine の状態を確認したいときは、Alpine DevTools という拡張機能があります ― 開発時に大いに役立ちます。

## モーダルに現在の手順のリストを表示する

入力欄の下には、現在の手順（編集時）か手順のリスト（新規作成時）を表示します。
これはリンクで使った UI と基本的には同じです：削除機能のついたリスト。

```html
<div class="mt-4" x-show="steps.length">
  <template x-for="(step, index) in steps" :key="index">
    <div class="flex items-center gap-3 p-2 bg-gray-50 rounded">
      <span class="flex-1" x-text="step"></span>
      <button type="button" @click="steps.splice(index, 1)" class="text-sm text-red-500">Remove</button>
    </div>
  </template>
</div>
```

モーダルフォームが送信されると、フォームの一部として手順の配列が送られます（Alpine が入力を配列に追加します ― 取得方法は以下のサーバセクションをご覧ください）。

## サーバ側の処理：手順のバリデーション

サーバでは、`steps` が配列であることとその要素が文字列（長さはお好きに）であることを検証します。
フォームリクエストを使っている場合はこれらのルールを追加します：

```php
// app/Http/Requests/StoreIdeaRequest.php (excerpt)
public function rules()
{
    return [
        'title' => ['required', 'string', 'max:255'],
        'description' => ['required', 'string', 'min:4'],
        'steps' => ['nullable', 'array'],
        'steps.*' => ['string', 'max:255'],
        // ...その他のルール...
    ];
}
```

これで 0 件以上の手順を送信できるようになります。

## 手順の保存（コントローラ）

手順は独自のテーブルで管理しアイデアとの関連を持たせるため、まずアイデアを作成してからリレーションを通じて手順を追加します。
典型的なパターンは以下のようになります。

リクエストの内容からアイデアを作成（手順は除く）し、その後、手順の配列をステップテーブルに合わせて整形して createMany を使います：

```php
// app/Http/Controllers/IdeaController.php (抜粋)
public function store(\App\Http\Requests\StoreIdeaRequest $request)
{
    // 検証済み属性からアイデアを作成（手順は除く）
    $attributes = $request->safe()->except('steps');
    $idea = auth()->user()->ideas()->create($attributes);

    // 手順があればそれを扱う
    $steps = $request->input('steps', []); // 文字列の配列
    if (!empty($steps)) {
        $rows = collect($steps)
            ->map(fn ($description) => ['description' => $description])
            ->toArray();

        $idea->steps()->createMany($rows);
    }

    return redirect()->route('ideas.show', $idea);
}
```

createMany には行ごとに手順テーブルの各カラムの値が入った配列の配列を渡します。
map で各文字列を `description` をキーとした配列にしているのはそのためです。

## リクエストの内容のデバッグ

送信されるフォームの内容がよく分からない場合は、dd() を使ってみましょう：

```php
dd($request->validated(), $request->input('steps'));
```

`steps` が文字列の配列となっているのが分かるでしょう。これをテーブルの行にマップしていきます。

## アイデアページで手順を表示する

アイデアページで具体的手順を表示するために、手順の有無を調べてから順に繰り返しましょう。

```html
{{-- resources/views/ideas/show.blade.php --}}
@if($idea->steps->isNotEmpty())
  <section class="mt-6">
    <h3 class="font-semibold mb-3">Actionable Steps</h3>

    @foreach($idea->steps as $step)
      <div class="flex items-center gap-3 p-3 mb-2 bg-white rounded shadow-sm">
        <form action="{{ route('steps.update', $step) }}" method="POST" class="inline">
          @csrf
          @method('PATCH')
          <button
            type="submit"
            role="checkbox"
            aria-checked="{{ $step->completed ? 'true' : 'false' }}"
            class="w-8 h-8 flex items-center justify-center rounded border"
            @if($step->completed) style="background:#10B981; color:white;" @endif
          >
            <!-- シンプルなチェックマーク -->
            {!! $step->completed ? '&#10003;' : '' !!}
          </button>
        </form>

        <span class="@if($step->completed) line-through text-gray-400 @endif">
          {{ $step->description }}
        </span>
      </div>
    @endforeach
  </section>
@endif
```

各手順の完了状態を更新するのにそれぞれで小さなフォームを使います（分かりやすくシンプルに）。
AJAX/Alpine で行うこともできますが、手順ごとのフォームで十分ですし使いやすいです。

## 手順の切替を行うルートとコントローラ

`completed` のブール値を切り替えるルートとコントローラを追加しましょう：

```php
// routes/web.php
Route::patch('/steps/{step}', [App\Http\Controllers\StepController::class, 'update'])
    ->name('steps.update')
    ->middleware('auth');
```

コントローラです：

```php
// app/Http/Controllers/StepController.php
public function update(\Illuminate\Http\Request $request, \App\Models\Step $step)
{
    // 認可：ユーザがこの手順を更新できるか調べる（ポリシーかゲートによって）
    $this->authorize('update', $step);

    $step->update(['completed' => ! $step->completed]);

    return back();
}
```

これで完了状態を切り替え、アイデアページにリダイレクトで戻ります。

## 表示のコツ

`object-cover` を使用してサムネイル（アイキャッチ画像も追加した場合）表示したときの高さを調整し、`overflow-hidden` と `rounded-t-lg` を使って画像がカードに付属しているように見せましょう。
手順ではチェック欄の高さを固定し、 完了時に `line-through` をセットし文字をくすんだ色にして見た目からでも状態が分かるようにします。

## 補足と次のステップ

- コントローラの中身が複雑になってきたら、手順の作成処理を action/service に移すことができます。そうするとテストや再利用が簡単になります。
- 既存のアイデアの編集を行う場合は、同じモーダルを再利用して Alpine の配列に既存の手順をセットするようにしてみましょう；サーバ側でも更新／削除をサポートする必要があります（削除ではレコードと必要に応じてファイルの削除を行います）。
- 完了状態の切替をクライアント側で行う（全画面更新をしない）場合は、トグルボタンを Alpine + フェッチ／AJAX リクエストに変更し、必要なところだけ DOM を更新してください。

これでエンドツーエンドの具体的手順を加えることができるようになります：追加／削除の UI、バリデーション、リレーションからの保存、完了状態の切替です。
必要であれば、コントローラや CLI やテストで呼び出し可能な、アイデアの追加処理（アイデア、手順、画像の保存、通知の発出）をまとめたアクションクラスを用意することができます。
