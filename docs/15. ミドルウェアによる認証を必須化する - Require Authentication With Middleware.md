# [ミドルウェアによる認証を必須化する](https://laracasts.com/series/laravel-from-scratch-2026/episodes/15)

アイデアのプライバシーと所有権について話しましょう — 各アイデアをユーザーに紐づけ、ミドルウェアでルートを保護し、ファクトリーと Tinker で素早くテストデータを作成し、ユーザーが自分のアイデアだけを見られるようにフィルタリングします。
これらは小さな変更ですが、実際の運用で正しく機能するシステムを実現するための重要なステップです。

## アイデアにユーザーとのリレーションを付与する

自分のアイデアと他人のアイデアを区別するには、各アイデアを特定のユーザーに紐づけなければいけません。
マイグレーションに外部キーを追加して、ユーザーを削除したらそのユーザーのアイデアも削除されるよう制約を設けましょう（孤児は作りません）。

マイグレーションはこのようになります：

```php
// database/migrations/xxxx_xx_xx_create_ideas_table.php
$table->foreignId('user_id')->constrained()->cascadeOnDelete();
```

`constrained()` で users テーブルを参照し、`cascadeOnDelete()` でユーザーが削除されたときにそのアイデアの一緒に削除されるようにしています。
こうしてデータベースを矛盾のない状態に保ちます。

## auth ミドルウェアでアイデアのルートを保護する

ログインしなければアイデアを作成したり更新したりできないようにしなければいけません。
そうするためにはミドルウェアが使用できます。

> ミドルウェアは玉ねぎの層のようなものです ― リクエストはアプリに到達するまでに各層を通ります。

ログインが必要なルートに `auth` ミドルウェアを適用します

```php
// routes/web.php

Route::middleware(['auth'])->group(function () {
    Route::get('/ideas', [IdeaController::class, 'index'])->name('ideas.index');
    Route::post('/ideas', [IdeaController::class, 'store'])->name('ideas.store');
    Route::get('/ideas/{idea}/edit', [IdeaController::class, 'edit'])->name('ideas.edit');
    Route::put('/ideas/{idea}', [IdeaController::class, 'update'])->name('ideas.update');
    Route::delete('/ideas/{idea}', [IdeaController::class, 'destroy'])->name('ideas.destroy');
});
```

ゲストがこれらのルートにアクセスすると、Laravel はデフォルトで `login` と名付けられたルートにリダイレクトさせます。
別の場所にリダイレクトさせたい場合は bootstrap フォルダ内の設定で変更できます。

## ゲスト用ルート：ログインユーザーにはユーザー登録させない

逆に、認証済のユーザーにはユーザー登録やログイン画面を見せたくありません。
そのようなルートには `guest` ミドルウェアを使用します。:

```php
Route::middleware(['guest'])->group(function () {
    Route::get('/register', [RegisteredUserController::class, 'create'])->name('register');
    Route::post('/register', [RegisteredUserController::class, 'store']);
    Route::get('/login', [SessionsController::class, 'create'])->name('login');
    Route::post('/login', [SessionsController::class, 'store']);
});
```

これによりユーザーエクスペリエンスが適切に保たれます：すでにログインしていれば、ゲスト用のページからリダイレクトではじかれます。

## 名前付きルートとログインへのリダイレクト

Laravel の auth ミドルウェアは `login` と名付けられたルートを探します（こうしてゲストのリダイレクト先を決めます）。
ですので、自身のログインページの名前を `login` とするか、設定でデフォルトのリダイレクト先を変更してください。

ルートに名前をつける場合は：

```php
Route::get('/login', [SessionsController::class, 'create'])->name('login');
```

こうすると `auth` ミドルウェアは、URI が変更になったとしても `route('login')` にリダイレクトさせることができます。

## ファクトリーと Tinker で素早くテスト用ユーザーを作成する

開発中にダミーのユーザーを作成すると、ユーザーごとにアイデア異なるという振る舞いを確認できるので便利です。

Tinker ではこのように作成します:

```bash
php artisan tinker
>>> App\Models\User::factory()->create(); // ユーザーを作成して保存する
```

これで、DB の GUI や別のファクトリーなどでアイデアを作成するときにユーザー ID を紐づけることができます。

```php
// アイデア作成時に新たにユーザーを用意して紐づける
>>> $idea = App\Models\Idea::factory()->for(App\Models\User::factory())->create();
```

または DB ツールを使ってアイデアの `user_id` を手動で更新し、別のユーザーのアイデアとします。

## 現在のユーザーのアイデアだけを表示するようにフィルタリングする

常に全アイデアを取得しないでください ― インデックスを認証済みユーザーに制限します。

シンプルなクエリーによる方法はこのようになります：

```php
// app/Http/Controllers/IdeaController.php
public function index(Request $request)
{
    $ideas = \App\Models\Idea::where('user_id', $request->user()->id)->get();

    return view('ideas.index', compact('ideas'));
}
```

ユーザーモデルにリレーションを張っている場合は、こちらの方が一般的です：

```php
public function index(Request $request)
{
    $ideas = $request->user()->ideas()->get();
    return view('ideas.index', compact('ideas'));
}
```

どちらの場合でも重要なのは：現在の認証済みユーザーに紐づいたアイデアだけを返すことです。


## 画面上ではどのように見えるか

適切にフィルタリングすれば、リストに表示されるのは自分のアイデアだけです。
ファクトリーで別のユーザーを作成してそのユーザーにアイデアを追加しても、そこで追加したアイデアは自分がログインしたときには見ることができません ― 適切なプライバシーを保つ方法です。

## 確認項目

- アイデアのマイグレーションに外部キー `user_id` を付与し、`constrained()->cascadeOnDelete()` をつける
- `php artisan migrate:fresh` を実行する（開発初期段階では）か、既に運用中であれば追加のマイグレーションを作成する
- アイデア関連のルートは `auth` ミドルウェアで保護する
- ユーザー登録とログインは `guest` ミドルウェアで保護する
- ログインルートには `login` と名前をつける（またはリダイレクト先を変更する）
- ファクトリーや Tinker を使ってテスト用のユーザーを作成し、アイデアの所有者に関するシナリオをテストする
- `auth()->id()` や `$request->user()->ideas()` によってインデックスをフィルタリングする

これらの小さな変更はエンドポイントを保護し、実際にアプリをマルチユーザー対応させます。
