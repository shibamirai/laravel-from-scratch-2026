# キューはいつ使うのか

キューイングされたジョブについて話しましょう ― それは何か、なぜ必要か、ローカルでどうやって動かすのかについてです。
通知とメールはすでにやりました；今度はもう少し深入りし、時間のかかる処理（たくさんのメールを送信するなど）は裏で実行してユーザを待たせないようにしましょう。

> キューは実行されるべき処理を貯めておく場です。

後でやらなければならない溜まった仕事だと考えましょう。

> ジョブとは"すべきこと"です。

これは仕事の単位を表します：メールを送る、状態を計算しなおす、このレコードのインデックスを付け替えるなどです。

> ワーカはキューからジョブを取り出し、それを実行するプロセス（この比喩では人）です。

ワーカはキューからジョブを取り出し、そのジョブのコードを実行します。

## なぜキューを使うのか

アイデアを作成したときにメールを一通送るだけであれば ― 同期していても構いません。
でももし 50 人のチーム全員に通知しなければならなかったり、いくつか大変なバックグラウンドタスク（状態の再計算や重い処理など）をしなければならなかったら、それらがすべて終わるまでユーザを待たせるのはよくありません。
キューは仕事をキューに割り当てたらすぐにユーザに応答し、ワーカはそのジョブを裏で実行します。

## 通知とメールのキューイング

Laravel では簡単です：Notification や Mailable に ShouldQueue インターフェースを実装すれば、Laravel はそれをインラインで実行する代わりに自動でキューに入れます。

Notification ではこのように実装します：

```php
<?php

namespace App\Notifications;

use Illuminate\Notifications\Notification;
use Illuminate\Contracts\Queue\ShouldQueue;
use Illuminate\Notifications\Messages\MailMessage;

class IdeaPublished extends Notification implements ShouldQueue
{
    public function via($notifiable)
    {
        return ['mail'];
    }

    public function toMail($notifiable)
    {
        return (new MailMessage)
            ->line('You published a new idea')
            ->action('Read Idea', url('/ideas/' . $this->idea->id));
    }
}
```

Laravel は `implements ShouldQueue` を見つけると、その場でメールを送信する代わりに通知を選択されたキューに入れます。

## キューのバックエンドと設定

デフォルトの状態でいろいろなキュードライバーを使えるようになっています：データベース、Redis、sync（同期）、その他にも多くのものがサードパーティーから提供されています。
バックグラウンド処理には `sync` は使わないでしょう。
.env ではドライバに `database` か `redis` をセットします。

```yaml
# .env (example)
QUEUE_CONNECTION=database
```

`database` ドライバを選択した場合はジョブテーブルが必要です。
このようにして作成しマイグレートすることができます：

```bash
php artisan queue:table
php artisan migrate
```

ジョブをキューに入れてからワーカが取り出すまでの間、`jobs` テーブルにレコードが追加されているのが分かるでしょう。

## ワーカをローカルで実行する

ワーカはキューにたまったジョブを実際に実行するものです。
ローカルではこのように実行します：

```bash
php artisan queue:work
```

このコマンドにより、キューにたまったジョブの処理を開始します。
キューに通知やジョブを追加しても、受信箱（またはログ）は変わりません：キューにジョブを追加したら、それを実行するためにワーカを起動させなければいけません。
ジョブをキューに追加しても、ワーカが起動してなければいつまでもそのままです。

複数のワーカを起動（別のターミナルで実行するなど）すれば、ジョブを並行して処理できます。

## ジョブの作成

キューに入れられるのは通知やメールだけではありません。
時間のかかる処理（状態の再計算、たくさんのメッセージの送信、重い IO）があれば、それ用のジョブを作りましょう。

```bash
php artisan make:job UpdateIdeaStatistics
```

これにより `app/Jobs` にクラスが作られます。
キューに入れるジョブには `ShouldQueue` インターフェースを実装し、行うべき仕事を `handle()` メソッドに記述します：

```php
<?php

namespace App\Jobs;

use Illuminate\Contracts\Queue\ShouldQueue;
use Illuminate\Bus\Queueable;
use Illuminate\Queue\SerializesModels;

class UpdateIdeaStatistics implements ShouldQueue
{
    use Queueable, SerializesModels;

    public function handle()
    {
        logger('UpdateIdeaStatistics job is being processed.');
        // ここに時間のかかる処理を記述
    }
}
```

コードまたは Tinker からジョブを割り当てます。

```php
// コードで割り当てる
UpdateIdeaStatistics::dispatch($idea);

// または Tinker から割り当てる
php artisan tinker
>>> App\Jobs\UpdateIdeaStatistics::dispatch($idea);
```

"割り当てる"とはジョブをキューに入れるだけです；動作中のワーカがそれを取り出し `handle()` を実行します。

## デバッグとログ

キューに入れられたジョブをデバッグするには、ジョブの `handle()` にログを入れましょう。
ログは `storage/logs/laravel.log` に出力されます。
ジョブを割り当てたのにログが出力されないというときは、ワーカが動いているかとキューコネクションの設定が適切かどうかを確かめましょう。

## ローカル環境でのワークフローのまとめ

ローカル環境でエンドツーエンドのテストを行うには：

1. QUEUE_CONNECTION が `sync` 以外になっていることを確認（`database` がお手軽です）
1. マイグレーションを実行し `jobs` テーブルを作成：`php artisan queue:table && php artisan migrate`
1. 通知／ジョブをキューに入れる（アプリまたは `Tinker` から行う）
1. ワーカを起動：`php artisan queue:work`
1. ログ、Mailpit（または自分のメールキャプチャツール）、ジョブテーブルで実行を確認

## ベストプラクティス

基本的に通知やメール送信はキューに入れます。特別な理由がない限り、レスポンスが速いままとなるように仕事はキューに割り当てましょう。
Notification/Mailable で `ShouldQueue` インターフェースを実装するか、重い仕事は明示的にジョブに割り当てます。
運用環境ではキュードライバーには堅牢なもの（Redis、SQS など）を使用し、スループットを高めたければ複数のワーカを動かしましょう。

最終的な目的はシンプルです：時間のかかるタスクをバックグラウンドで実行することによって、**ユーザのためにパフォーマンスと応答時間を改善しましょう**。
もしバックグラウンドでできることがあるのであればキューに入れます。
必要であれば、キューに入れる通知、ジョブ、キューの設定、それらをローカル環境で実行するコマンドをまとめた最小限のリポジトリをお渡しできます。
