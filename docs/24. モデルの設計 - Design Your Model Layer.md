# [モデルの設計](https://laracasts.com/series/laravel-from-scratch-2026/episodes/24)

UI に取り掛かる前にドメインとデータベースの構造について話しましょう。
私は最初にモデル、マイグレーション、ファクトリ、リレーションを用意することを好みます。
それらをバックボーンとして残りのアプリを構築できるからです。

## モデルとそれをサポートするファイルの作成

まずはターミナルを開き、必要なものを作成するところから始めます：モデル、マイグレーション、ファクトリ、フォームリクエスト、ポリシー、そしてコントローラです。
これらはいくつかの artisan コマンドで実行できます。
例えば：

```bash
php artisan make:model Idea -m -f
php artisan make:request StoreIdeaRequest
php artisan make:policy IdeaPolicy --model=Idea
php artisan make:controller IdeaController --resource
```

これによりモデル、編集用のマイグレーション、テストデータのためのファクトリ、入力を検証するリクエスト、アクションを制限するポリシー、そして CRUD を実装するリソースコントローラが作成されます。

## アイデアのマイグレーション

アイテムには何が必要かを考えましょう。
タイトルは必須にして他のフィールドは任意にしましょう。
リンクは JSON として保存、手順はそれぞれのテーブルに保存し、各アイデアはユーザに紐づけます。

マイグレーションはこのようになります：

```php
public function up()
{
    Schema::create('ideas', function (Blueprint $table) {
        $table->id();
        $table->string('title');
        $table->text('description')->nullable();
        $table->foreignId('user_id')->constrained()->cascadeOnDelete();
        $table->json('links')->default('[]');
        $table->string('status')->default('pending'); // enum にもできるが文字列で構わない
        $table->string('image_path')->nullable();
        $table->timestamps();
    });
}
```

修正したらマイグレーションを実行します：

```bash
php artisan migrate
```

## ステップテーブル (別のモデルとして作成)

手順（ステップ）はアイデア実現にむけて行うべき具体的な TODO なので、別テーブルで扱います。
ステップテーブルはシンプルです。

```php
public function up()
{
    Schema::create('steps', function (Blueprint $table) {
        $table->id();
        $table->foreignId('idea_id')->constrained()->cascadeOnDelete();
        $table->string('description');
        $table->boolean('completed')->default(false);
        $table->timestamps();
    });
}
```

再びマイグレーションを実行してテーブルを作成します。

## 言葉の意味

> ステップはアイデアに紐づけられた実際のタスクです。

それらは小さな個々のアイテムで（これをして、次はあれ）、各ステップはある１つのアイデアに紐づいています。

## モデルの設定とキャスト

アイデアモデルでは、データベースから取り出したデータを PHP で扱いやすくするためにキャストが必要です。
リンクは配列で欲しいですし、ステータスには enum を使います。
キャストの例：

```php
class Idea extends Model
{
    protected $casts = [
        'links' => 'array',
        'status' => \App\Enums\IdeaStatus::class, // enum にキャスト
    ];

    protected $attributes = [
        'status' => \App\Enums\IdeaStatus::Pending->value, // Eloquent でのデフォルト値
    ];
}
```

注意：モデルにセットするデフォルトステータスは、ファクトリで作るときと実際のインスタンスのデフォルト値がデータベースに保存する前から同じになるようにすること。

## アプリの初期起動処理: 便利な Eloquent 設定

安全に開発を進めるために AppServiceProvider::boot にいくつか手を加えます：マスアサインメントのガードを外し（何をアサインするか明示的に管理できます）、Eloquent の振る舞いを厳密にして問題を早めに見つけられるようにし、開発時には遅延ロードを禁止して N+1 問題の発生を分かりやすくします。

このような設定になります：

```php
use Illuminate\Database\Eloquent\Model;

public function boot()
{
    Model::unguard();

    if ($this->app->isLocal()) {
        Model::preventLazyLoading(true);
        Model::preventAccessingMissingAttributes(true);
    }
}
```

この設定によって問題に早くから気づけるようになります（そして環境によって設定を変更できます）。

## ステータスの Enum

ステータスの文字列をコード中にばらまくのではなく、enum で定義します。

```bash
php artisan make:enum IdeaStatus --backed
```

<p>enum の例：</p>

```php
enum IdeaStatus: string
{
    case Pending = 'pending';
    case InProgress = 'in_progress';
    case Completed = 'completed';

    public function label(): string
    {
        return match($this) {
            self::Pending => 'Pending',
            self::InProgress => 'In Progress',
            self::Completed => 'Completed',
        };
    }
}
```

これでステータスの値は型付けされ、UI のコードでは ->label() のようにしてメソッドで呼び出せます。

Tinker で素早く試すことができます：

```bash
php artisan tinker
>>> $idea = new \App\Models\Idea;
>>> $idea->status = \App\Enums\IdeaStatus::InProgress;
>>> $idea->status->label();
```

## リレーション

ドメインが自然に読めるようにリレーションを張ります。

アイデアモデルでは：

```php
public function user()
{
    return $this->belongsTo(\App\Models\User::class);
}

public function steps()
{
    return $this->hasMany(\App\Models\Step::class);
}
```

ステップモデルでは：

```php
public function idea()
{
    return $this->belongsTo(\App\Models\Idea::class);
}
```

ユーザモデルでは：

```php
public function ideas()
{
    return $this->hasMany(\App\Models\Idea::class);
}
```

こうしてドメイン同士を関連づけます：ユーザは複数のアイデアを持ち、アイデアはそれぞれ1人のユーザに属し、アイデアには複数のステップがあり、各ステップは１つのアイデアに属します。

## ファクトリとデフォルトデータ

ファクトリで適切なデフォルトデータを追加し、素早くリアルなテストデータを用意できるようにしましょう。

アイデアファクトリ（抜粋）：

```php
'user_id' => \App\Models\User::factory(),
'title' => fake()->sentence(),
'description' => fake()->paragraph(),
'links' => ['https://laracasts.com'],
// status default comes from the model attributes
```

ステップファクトリ（抜粋）：

```php
'idea_id' => \App\Models\Idea::factory(),
'description' => fake()->sentence(),
'completed' => false,
```

ステータスは、モデルの $attributes でデフォルト値をセットしているので、ファクトリ内で値をセットしなくてもファクトリで作られたアイデアには正しいデフォルト値がセットされます。

## テスト：リレーションが正しいか確かめる

リレーションやデフォルト値が期待通りになっているか、簡単なテストを書いて確かめましょう。
テストが毎回クリーンな状態で実施されるように RefreshDatabase を使用します。

例 (PHPUnit / Pest での書き方)：

```php
public function test_idea_belongs_to_user()
{
    $idea = Idea::factory()->create();
    $this->assertInstanceOf(\App\Models\User::class, $idea->user);
}

public function test_idea_has_empty_steps_by_default()
{
    $idea = Idea::factory()->create();
    $this->assertTrue($idea->steps->isEmpty());
}

public function test_adding_a_step_to_idea_persists()
{
    $idea = Idea::factory()->create();
    $idea->steps()->create(['description' => 'Do the thing', 'completed' => false]);
    $this->assertCount(1, $idea->fresh()->steps);
}
```

カラムがないとかデフォルト値が異なるといってテストに失敗したら、テストで指示されたことに従ってください。
細かいところはこのテストによって実装してもらいたいので、あえてテストに失敗する状態にしています ― 例えば、ステップの completed カラムはマイグレーションでデフォルトで false にしていなければいけない、などです。

## ちょっとしたワークフローのコツ

テストを素早く実行できるように Better Pest のようなエディタの拡張機能（またはキーバインド）を導入することをお勧めします。
私はカーソルをテストの中に置いて、修正中はそのテストだけを実行できるようにしています。

## まとめ

アイデアとステップのモデルを作成し、マイグレーションとファクトリを作成し、適切なキャストとステータスの enum を追加し、テストで基本的なリレーションを確認しました。
こうして信頼できるドメインモデルが出来上がったので、安心して UI やコントローラの構築を始めることができます。
次のエピソードでは UI に取り掛かります ― 表示ページ、フォーム、編集モーダルなど ― ただし、現時点でデータベースとモデルは十分に整備されており、反復開発が可能な状態になっています。
