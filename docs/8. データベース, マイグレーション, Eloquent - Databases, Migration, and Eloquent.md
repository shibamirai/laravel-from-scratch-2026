# データベース, マイグレーション, Eloquent

アプリを実際のデータベースに移行する話をしていきましょう — マイグレーションの作成、実行、そして DB ファサードと Eloquent の両方を使った読み書きについてです。
ここでアプリは見せかけを止め、実際にデータを永続化します。
これは非常に満足感を得られる作業だとお約束します。

## データベース構成と SQLite

Laravel はデフォルトでローカル開発用に SQLite（ファイルベースの DB）を採用しています。
接続設定は .env ファイルで確認できます。
.env ファイルはプライベートな環境設定ファイルと考えてください：ローカル開発用と本番環境用の値を別々に設定します。

SQLite を使用している場合、DB ファイルは通常 database/database.sqlite に存在します。
作業中にテーブルや行を確認するには、GUI（TablePlus や DB Browser for SQLite がおすすめです）でこのファイルを開くことができます。

## マイグレーション（データベースのバージョン管理）

> マイグレーションとはデータベースのバージョン管理と考えてください。

マイグレーション機能により、スキーマをプログラムで記述できるため、チームメンバーは変更を共有し、変更を適用または元に戻すことができます。
テーブルの作成、列の追加、列の削除 — これらすべてを Git にコミット可能なコードで実現します。

初期のマイグレーションを作成するために実行したコマンドは以下の通りです：

```bash
php artisan make:migration create_ideas_table
```

生成されたファイルを開くと、`up()` メソッド（マイグレーション実行時の処理）と `down()` メソッド（元に戻す方法）が確認できます。
Laravel は `up()` で作成したテーブルを `down()` で自動的に削除しますが、詳細な `down()` ロジックを記述するか、フォワード専用マイグレーションを好むかは自由です。

## アイデアテーブルの作成

このマイグレーションでは、アイデアの説明にはテキスト列を選択しました（これにより 255 文字の制限を受けません）。
マイグレーションスニペットの例：

```php
<?php

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

class CreateIdeasTable extends Migration
{
    public function up()
    {
        Schema::create('ideas', function (Blueprint $table) {
            $table->id();
            $table->text('description');
            $table->string('state')->default('pending');
            $table->timestamps();
        });
    }

    public function down()
    {
        Schema::dropIfExists('ideas');
    }
}
```

後から列を追加する必要がある場合、初期開発段階であれば元のマイグレーションを修正して `php artisan migrate:refresh` を実行するか、列の追加/削除を行う新しいマイグレーションを作成します（共同プロジェクトでは推奨）。

列を追加する新しいマイグレーションを作成するには：

```bash
php artisan make:migration add_state_to_ideas_table --table=ideas
```

次に、新しいマイグレーションの up() でカラムを追加し、down() で削除します。

## マイグレーションの実行と更新

以下のコマンドでマイグレーションを実行してください：

```bash
php artisan migrate
```

初期開発段階でマイグレーションを変更し、データの損失を気にしない場合、`migrate:refresh` はすべてのマイグレーションをロールバックし、再度実行します：

```bash
php artisan migrate:refresh
```

注意：リフレッシュするとデータが切り捨てられます。
より安全でチームに優しい方法は、スキーマを変更する新しいマイグレーションを作成し、`php artisan migrate` を実行することです。
チームメンバーはマイグレーションをプルし、ローカルで実行できます。

## データベースの確認

SQLite ファイルを GUI（TablePlus、DB Browser for SQLite など）で開いてテーブルと行を確認してください。
テスト中に手動で挿入することも可能です。
各アイデアには自動増分ID、説明、状態、タイムスタンプが設定されます。

## DB ファサードを用いたデータベースへのクエリ実行

素早く生のクエリが必要な場合は、DB ファサードを使用してください：

```php
use Illuminate\Support\Facades\DB;

$ideas = DB::table('ideas')->get(); // stdClass オブジェクトのコレクションを返す
```

Laravel は、ルートからコレクションを返す場合、それらを JSON にシリアライズします。

各行はオブジェクトのようにアクセスできます:

```php
$first = $ideas->first();
echo $first->description;
```

しかし、より洗練されたモデルベースの API のためには Eloquent の方がいいでしょう。

## Eloquent: Laravel の ORM

Eloquent はデータベーステーブルを PHP クラスにマッピングします。
モデルを作成するには：

```bash
php artisan make:model Idea
```

これにより、app/Models/Idea.php が生成されます。
これで、そのモデルを介して ideas テーブルを操作できるようになります。

すべてのアイデアを取得する：

```php
use App\Models\Idea;

$ideas = Idea::all(); // Idea モデルのコレクション
```

アイデアを１つ取得する：

```php
$idea = Idea::find(1); // Idea モデルか null を返す
```

条件付きクエリ：

```php
$pending = Idea::where('state', 'pending')->get();
```

条件を追加する場合は、get()（または first()、paginate() など）を呼び出すことを忘れないでください — クエリビルダーのチェーンを構築していることになります。

## Eloquent での挿入

Eloquent で新しいアイデアを作成するには：

```php
$idea = Idea::create([
    'description' => 'Launch a new Laracasts course',
    'state' => 'pending',
]);
```

ガード付き属性の警告が発生した場合、Eloquent はデフォルトで一括代入を保護しています。
モデルの $fillable を明示的に設定するか、ローカルでの簡単な実験であればガードを無効化できます：

```php
// app/Models/Idea.php
class Idea extends Model
{
    protected $guarded = []; // ガードの無効化 (注意して使用すること)
}
```

Eloquent は作成日時と更新日時も自動的に処理します。

## フィルタリングと条件付きクエリ

リクエスト入力をチェックすることで、クエリを条件付きかつ柔軟にできます。
クエリ文字列によるオプションのフィルタリングを有効にするために私が使用するパターンは次のとおりです：

```php
use Illuminate\Http\Request;

public function index(Request $request)
{
    $ideas = Idea::when($request->has('state'), function ($query) use ($request) {
        $query->where('state', $request->state);
    })->get();

    return view('ideas.index', compact('ideas'));
}
```

このコードを日本語に翻訳すると：リクエストに 'state' が含まれている場合、その 'state' でフィルタリングしてから結果を取得する、そうでない場合は全てのアイデアを返す、となります。

## 実用的な注意事項

開発初期段階でスキーマを頻繁に変更したい場合は、編集してから `migrate:refresh` で問題ありません。
チーム環境や本番環境では、追加型マイグレーション（スキーマ変更ごとに新規マイグレーションを作成）を推奨します。
これにより全員が `php artisan migrate` を実行し、同一のデータベース構造を取得できます。

ローカルでのプロトタイピングには SQLite を使用しましょう。
設定不要で高速です。
本番環境に移行する際は、同じマイグレーションコードで MySQL や PostgreSQL などに切り替えることになるでしょう。
