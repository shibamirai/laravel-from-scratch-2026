# [認可は必須要件](https://laracasts.com/series/laravel-from-scratch-2026/episodes/38)

認可について話しましょう ― まず簡単にコードの整理を行ってから、次にポリシーとミドルウェアを設定して、正しい権限を持つユーザだけがアイデアの閲覧、更新、削除を行えるようにします。
簡単なコードレビューで見つけたことや、レビューで見つかったバグ、それらをテストで修正する方法、それからポリシー（とミドルウェア）を使用してコントローラアクションを制限する方法をお見せします。


## 簡単なコードレビューとそこで見つかったバグ

認可を追加する前に簡単な静的レビューを実施しました（CodeRabbit を使用）。
それによりアクションクラスの１つで問題が発覚しました：トランザクションクロージャの中で未定義の変数が参照されていました。
簡単に説明すると、クロージャ内で $attributes を参照していたんですがそれを使用可能にしていなかったため、クロージャが常に真偽を誤って、手順の追加がされていませんでした。

“手順が追加されること” のチェックをテストに入れていなかったため、テストでは見つけることができませんでした ― そのため実際には黙って何もしていなかったにも関わらす、アクションが正常に動作しているように見えていました。
これこそがコードレビューとテストの両方をやらなければいけない理由です：レビューが怪しいところをあぶりだし、テストで意図を確かめます。

アクションを修正した後、ブラウザテストを拡張して手順が実際に作成されたことを確認するようにしました。
私が追加したテストの重要な部分がこちらです（Pest スタイル）：

```php
<?php

it('creates steps when submitted', function () {
    // ... ログインして、アイデア作成モーダルを開き、フィールドに入力 ...

    // UI から手順を２つ追加
    $browser->click('@add-step-button')->type('@step-input', 'First step')->click('@add-step-submit');
    $browser->click('@add-step-button')->type('@step-input', 'Second step')->click('@add-step-submit');

    // アイデアを送信
    $browser->click('@create-idea-submit');

    // データベースでアイデアに手順が２つあることを確認
    $idea = \App\Models\Idea::latest()->first();
    expect($idea->steps->count())->toBe(2);
});
```

テストに失敗すれば、それはバグがあるということです。
コードを修正してからテストを再実施し、すべて緑になるようにしましょう。
私はいつもそのようにしています。

## ポリシー：認可のロジックを１か所に集める

ポリシーはリソースの認可ルールをまとめておくのに適した場所です。
アイデアモデルを生成したとき、同時にポリシーも生成しました。
ポリシーはシンプルにしておきましょう：現在のユーザはこのアイデアを操作できますか？

ポリシーの例 (app/Policies/IdeaPolicy.php)：

```php
<?php

namespace App\Policies;

use App\Models\Idea;
use App\Models\User;

class IdeaPolicy
{
    public function manage(User $user, Idea $idea): bool
    {
        // アイデアの持ち主だけが操作（更新／削除）できる
        return $idea->user_id === $user->id;
    }
}
```

この能力には `manage` と名付けましたが、扱っている内容にフィットする名前であればどんなものでも (update, modify, workWith) 構いません。
このポリシーは対象のアイデアと現在のユーザを受け取り、判定します。

ポリシーを手動で作成したときはそれを AuthServiceProvider に登録するのを忘れないようにしましょう。

## コントローラまたはルートからポリシーを利用する

認可チェックはこの２か所でよく行われます：コントローラアクション内またはルートのミドルウェアです。
チームでどちらかを決めましょう。

コントローラでの認可（アクション内で明示され分かりやすい）：

```php
public function update(Request $request, Idea $idea)
{
    $this->authorize('manage', $idea);

    // 更新の検証ずみ...
}
```

<p>Route/middleware で行う場合 (ルートとあわせて意図を定義する):</p>

```php
Route::get('/ideas/{idea}', [IdeaController::class, 'show'])
    ->middleware(['auth', 'can:manage,idea']);
```

どちらも中身は同じです。
ミドルウェアの方が簡潔（ `can:やること、パラメータ名` と読めるでしょう）ですが、コントローラで呼び出す方が明示的でコード検索で見つけやすいです。
分かりやすさの観点から私はコントローラの方を選ぶことが多いですが、どちらも適切です。


## なぜ認可のテストを行うのか

認証と認可の両方をテストしなくてはいけません：

> 未認証ユーザはログインにリダイレクトされること。
> 認証済みだが持ち主ではないユーザには 403(forbidden) を表示すること。
> 持ち主のユーザは自分のリソースにアクセスし修正できること。

閲覧ページの表示に関するテストの例（Pest スタイル）：

```php
it('requires authentication to view an idea', function () {
    $idea = Idea::factory()->create();
    $this->get(route('ideas.show', $idea))->assertRedirect(route('login'));
});

it('forbids viewing an idea the user does not own', function () {
    $user = User::factory()->create();
    $idea = Idea::factory()->create(); // 他人のアイデア

    actingAs($user);
    $this->get(route('ideas.show', $idea))->assertForbidden();
});
```

これらのテストで期待される振る舞いを確認します。
誰かがポリシーを削除したり壊したりすればテストが失敗するようになるでしょう。


## 実践的な補足事項

- ポリシーは複数のルールがある（閲覧と更新と削除）場合に最適です。
- 認可をルートで簡潔に定義するにはミドルウェア (`can:`) が便利です。
- 認可の振る舞いは必ずテストで確認しましょう ― 一見正しそうですがポリシーチェックをしていないコードを書いてしまいがちです。
- トランザクションの一環としてアクション内で関連モデル（手順など）を作成するときは、クロージャ内で参照する変数を使用可能にしておきましょう（クロージャで `use` 宣言するかロジックをクロージャの外に移動する）。テストがあればこのミスに気づけます。
- 全体的に認証ルールをつけたければ `auth` を適用したルートグループを使い、リソース個別のチェックが必要なところには `can:` を追加しましょう。

## まとめ

1. 静的レビューツールを使いましょう ― 怪しいところ（クロージャ内の未定義変数など）を指摘してくれます。
1. 失敗パターンがテストに含まれていなかったら、実際に想定される影響（今回の場合は "手順が作成されていること"）をテストに含めましょう。
1. 認可ロジックをポリシーにまとめ、それをコントローラ ($this->authorize(...)) またはルートのミドルウェア (`can:`) から呼び出しましょう。
1. 認証（ログインページにリダイレクト）と認可（403 forbidden）の両方をテストしましょう。

必要であれば、プロジェクトにそのままコピーして使える最小限のポリシーのファイルと `$this->authorize('manage', $idea)` を使ったコントローラの例、それに実際に使用した特定のテストをお渡しできます。
