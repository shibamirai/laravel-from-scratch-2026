# ポリシーによる認可

ポリシーについて話しましょう ― コントローラーでモデルに対する認可を行う方法です。
ゲートはルートレベルのちょっとしたルールに適していますが、ポリシーはモデル単位での判定（表示、更新、削除など）を行うための用意された場所を提供します。

## ポリシーとは何か

> ポリシーはゲートのコントローラー版です ― モデル固有の認可ルールを与えることができます。

ポリシーを使用すると、認証ロジックを保護対象のモデルの近くに配置でき、ビュー、コントローラー、ルート全体で再利用可能な名前付きメソッド（権限）を作成できます。

## ポリシーの作成

アイデアモデルに対するポリシーはこのように作成します。
make コマンドに自動で名前を付けさせることもできます。

```bash
php artisan make:policy IdeaPolicy --model=Idea
```

app/Policies/IdeaPolicy.php が作成され、同時にいくつかのスタブメソッド (view, create, update, delete, など) が用意されます。
すべてのメソッドを使う必要はありません ― 必要なものだけ実装します。

## 簡単な更新ルール

作成されたポリシーを開き、必要なロジックを実装します。
例えば、アイデアの作者だけが更新できるようにします。

```php
// app/Policies/IdeaPolicy.php

use App\Models\User;
use App\Models\Idea;
use Illuminate\Auth\Access\Response;

public function update(User $user, Idea $idea)
{
    // ブール値を返す
    return $user->id === $idea->user_id;

    // またはもっと細かく制御するためにアクセスレスポンスを返す：
    // return $user->id === $idea->user_id
    //     ? Response::allow()
    //     : Response::denyAsNotFound(); // 403 の代わりに 404 を返す
}
```

モデルには Eloquent インスタンス同士を比較する `is()` というヘルパーがありました。
これを使うと、同様のルールをこのように表現できます：

```php
public function update(User $user, Idea $idea)
{
    return $user->is($idea->user);
}
```

この方が分かりやすく、手動で ID を比較しなくて済みます。

## コントローラーでポリシーを使用する

コントローラーでポリシーをチェックするには authorize (または Gate::authorize) をコールします。
フレームワークは適切なポリシーを探して指定したメソッドを呼び出します。

```php
use Illuminate\Support\Facades\Gate;

public function show(Idea $idea)
{
    // コントローラーのヘルパー - 拒否時は AuthorizationException が発生
    $this->authorize('update', $idea);

    // または:
    Gate::authorize('update', $idea);

    return view('ideas.show', compact('idea'));
}
```

許可されなければ例外が発生し、適切な HTTP レスポンスに変換されます（デフォルトは 403, またはポリシーが返すもの）。

## ビューでポリシーを利用する

Blade には `@can` ディレクティブがあり、これは裏で同じポリシーを使います。

```html
@can('update', $idea)
  <a href="{{ route('ideas.edit', $idea) }}">Edit</a>
@endcan
```

例外ではなく別の方法で個別に対処したい場合は、PHP で `auth()->user=>can('update', $idea)` を呼び出すことができます。

## ルートレベルでの保護

ルートレベルでのチェックがお好みであれば、ルートに `can` ミドルウェアを適用します。
これはルートの実行前にポリシーを適用します。

```php
Route::get('/ideas/{idea}/edit', [IdeaController::class, 'edit'])
    ->can('update', 'idea'); // 'idea' はルートパラメータを参照している
```

このポリシーは、コントローラー内部から呼び出す代わりにミドルウェアから呼び出されているというだけで、まったく同じものです。
どちらを採用するかはチームによって異なります；どちらかを選んで一方に統一してください。

## モデルを必要としないアクション (Create など)


一部のメソッドはモデルを必要としません（例えばアイデアの作成）。
その場合は Laravel が使用するポリシーを見つけられるようにモデルクラスを参照します。

```php
// 作成画面のコントローラーでは
Gate::authorize('create', \App\Models\Idea::class);

// ルートでは:
Route::get('/ideas/create', [IdeaController::class, 'create'])
    ->can('create', \App\Models\Idea::class);
```

このポリシーメソッドの定義は `public function create(User $user)` のようになり（アイデアパラメータなし）、ブール値かアクセスレスポンスを返します。

## レスポンスコードの制御 (403 か 404 か)

エンドポイントの存在を隠したい場合は、forbidden ではなく not-found をポリシーから返します。

```php
use Illuminate\Auth\Access\Response;

public function update(User $user, Idea $idea)
{
    return $user->id === $idea->user_id
        ? Response::allow()
        : Response::denyAsNotFound(); // 403 の代わりに 404 を返す
}
```

認可されていないユーザーにはページの存在自体を隠したい場合に有効です。

## 実践的なアドバイス

小さなアプリであれば簡単なチェック（`isAdmin()` ヘルパーでユーザー ID か `role` カラムを使う）で十分であり、メンテナンスもしやすいです。
もし多くの役割や複雑なルールを扱うのであれば、ポリシーを構成し役割ベースか許認可のシステムを考えましょう。どちらかを選んだらそれに統一します。

必要なところでは必ず認可を忘れないでください：表示（間違ったものを表示しない）、ルート（エンドポイントを保護）、そしてコントローラー（認可の例外をカスタマイズしたい場合）など。
テストは意図しない情報漏洩を検出するのに役立ちます — UIによる非表示だけに頼らないようにしましょう。

まとめると：ポリシーを作成し、必要なメソッドを実装 (update, view, create, delete)、コントローラーかルートで認可し 403 か 404 のどちらかを返す。
次回はリソースの認可をもっと幅広く行ういくつかのパターンをお見せします。
