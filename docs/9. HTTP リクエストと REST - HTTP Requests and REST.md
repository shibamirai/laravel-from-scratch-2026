# HTTP リクエストと REST

リソースルート、レコードの表示と編集、そして CRUD を楽しく感じさせる Laravel の便利な小技について話しましょう。
find と findOrFail の比較、ルートモデルバインディング（いつもワクワクする部分）、メソッド偽装による編集、auth/guest ミドルウェアによるルート保護を扱います。
短く、実践的で、役立つ内容です。

## 表示、編集、更新、破棄 — リソースの形状

アイデア（または投稿、記事）のようなリソースを扱う場合、共通して必要となるエンドポイントがあります：一覧表示用の index、単一アイテム表示用の show、作成用の create/store、変更用の edit/update、削除用の destroy です。
これらは HTTP メソッドとコントローラーアクションに見事に対応します。

単一のアイデアを /ideas/{id} で表示したい場合、ルートワイルドカードとして id をキャプチャし、手動でモデルを検索できます：

```php
// routes/web.php
Route::get('/ideas/{id}', function ($id) {
    $idea = Idea::find($id); // 見つからなければ null を返す
    return view('ideas.show', ['idea' => $idea]);
});
```

ただし、コレクションを返す場合とレコードを１つ返す場合の違いに注意してください。
get() や first() などのメソッドは find() とは動作が異なります。

ルートから直接アイデアオブジェクトを返すと、Laravel はそれを JSON に変換します：

```php
Route::get('/ideas/{id}', function ($id) {
    return Idea::find($id);
});
```

これはアイデアの属性を JSON 形式でブラウザに出力します。

## 素早く失敗させる：findOrFail

レコードが存在しない場合、ビューに null を渡して後でエラーを発生させるよりも、404 で処理を中止したい場合がよくあります。
find() + 手動チェックの代わりに、findOrFail() を使用してください：

```php
$idea = Idea::findOrFail($id);
```

一致するレコードが存在しない場合、自動的に ModelNotFoundException がスローされ、404 レスポンスが返されます。

## ルートモデルバインディング（小さな魔法）

ルートモデルバインディングは定型コードを完全に排除します。
ルートのパラメータ名をコントローラ/クロージャのパラメータ名と同じにし、モデルの型ヒントを指定するだけで、Laravel が自動的にモデルを解決します：

```php
// routes/web.php
Route::get('/ideas/{idea}', function (Idea $idea) {
    // $idea は、指定されたIDに該当する Eloquent モデルインスタンスになっています
    return view('ideas.show', ['idea' => $idea]);
});
```

その ID に対応するアイデアが存在しない場合、Laravel は自動的に 404 を返します。
重要な点：ルートのワイルドカード名とパラメータ名は一致している必要があります（上記では両方とも `idea`）。

## 編集とメソッド偽装

編集ページにはアイデアのデータが事前入力されたフォームが表示されます。
更新フォームは PATCH（または PUT）リクエストを送信すべきですが、ブラウザは GET と POST のみをサポートしています。
Laravel は内部で POST を使用しながら意図した HTTP メソッドを通知するメソッド偽装を提供します。

編集フォームの例は以下の通りです：

```html
<!-- resources/views/ideas/edit.blade.php (抜粋) -->
<form method="POST" action="/ideas/{{ $idea->id }}">
  @csrf
  @method('PATCH') <!-- メソッド偽装 -->
  <label for="description">Description</label>
  <textarea name="description" id="description">{{ $idea->description }}</textarea>

  <button type="submit">Update</button>
</form>
```

ルート側では、その patch リクエストを処理します：

```php
Route::patch('/ideas/{idea}', [IdeaController::class, 'update']);
```

そしてコントローラーでは、ルートモデルバインディングを使用できるため、アイデアインスタンスが自動的に注入されます：

```php
public function update(Request $request, Idea $idea)
{
    $idea->update(['description' => $request->input('description')]);
    return redirect("/ideas/{$idea->id}");
}
```

（分かってます、バリデーションすべきですが、それは別途扱います — これは正常な処理の流れです。）

## リソースの削除

リソースを削除するには、DELETE リクエストに応答する必要があります。
同じメソッド偽装の手法が適用されます。
よくある落とし穴：フォーム内にフォームをネストできないため、削除用の別フォームを設置してください（または JavaScript/fetch で削除処理を行う方法も選択可能です）。

```html
<form method="POST" action="/ideas/{{ $idea->id }}">
  @csrf
  @method('DELETE')
  <button type="submit" class="btn btn-danger">Delete</button>
</form>
```

ルート：

```php
Route::delete('/ideas/{idea}', [IdeaController::class, 'destroy']);
```

そしてコントローラー：

```php
public function destroy(Idea $idea)
{
    $idea->delete();
    return redirect('/ideas');
}
```

## ミドルウェアによるルートの保護

通常、作成/編集/更新/削除ルートには認証を要求し、ログイン/登録ルートにはゲストの認証を要求します。
ミドルウェアが頼りになります。

> ミドルウェアは、アプリケーションの核心へと続く玉ねぎの層のようなものです。

サインインしたユーザーが必要なルートには `auth` を適用し、認証されていないユーザーのみにアクセスを許可したいルートには `guest` を適用します。

グループ保護されたルート：

```php
// routes/web.php
Route::middleware('auth')->group(function () {
    Route::get('/ideas', [IdeaController::class, 'index']);
    Route::get('/ideas/create', [IdeaController::class, 'create']);
    Route::post('/ideas', [IdeaController::class, 'store']);
    Route::get('/ideas/{idea}/edit', [IdeaController::class, 'edit']);
    Route::patch('/ideas/{idea}', [IdeaController::class, 'update']);
    Route::delete('/ideas/{idea}', [IdeaController::class, 'destroy']);
});

Route::middleware('guest')->group(function () {
    Route::get('/login', [AuthenticatedSessionController::class, 'create'])->name('login');
    Route::get('/register', [RegisteredUserController::class, 'create'])->name('register');
    // ...
});
```

Laravel はデフォルトで認証されていないリクエストを `login` という名前のルートにリダイレクトします（Authenticate ミドルウェアで設定）。
そのため、そのルートが存在することを確認するか、リダイレクト先を変更してください。

## 現在のユーザーのアイデアのみを表示する

アイデアが非公開の場合、Idea::all() を返さないでください — クエリを認証済みユーザーにスコープしてください。
単純な where を使用するか：

```php
$ideas = Idea::where('user_id', auth()->id())->get();
```

あるいは、User 上でリレーションを定義したら、次のようにできます：

```php
$ideas = auth()->user()->ideas()->get();
```

（リレーションについては次回取り上げます。）

## 小さな UX のコツ

画面サイズが小さい場合にナビゲーションが折りたたまれ「作成」リンクが非表示になる場合、ユーザーがより見つけやすいよう、インデックスページに小さな目立つ作成リンクを追加することを検討してください。
小さな UX の改善も重要です。

## 簡単なまとめ

リソース操作と HTTP メソッドを概念図として捉えましょう：index（GET /ideas）、show（GET /ideas/{idea}）、create/store（GET /ideas/create、POST /ideas/{idea}）、edit/update（GET /ideas/{idea}/edit、PATCH /ideas/{idea}）、destroy（DELETE /ideas/{idea}）。
コントローラー全体で手動で null チェックを避けるため、findOrFail やルートモデルバインディングを使用してください。
ルートは `auth` と `guest` ミドルウェアで保護します。
メソッド偽装によりフォームから PATCH/DELETE リクエストが可能になります。
作成されたリソースは常に `auth()->id()` または `auth()->user()->ideas()->create()` で認証済みユーザーに関連付けます。

必要であれば、ルートモデルバインディングと簡易バリデーションを使用した index/show/create/store/edit/update/destroy を備えたすぐに使える IdeaController のスケッチを作成できます。
