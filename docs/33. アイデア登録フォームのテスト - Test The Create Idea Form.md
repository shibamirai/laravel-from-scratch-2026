# [アイデア登録フォームのテスト](https://laracasts.com/series/laravel-from-scratch-2026/episodes/33)

アイデア登録モーダルの自動化について話しましょう。
見た目も整え、機能も実装しましたが、手動でのテストでは実際に問題が発生しないとそれを見つけることができません。
この動画ではその手順をプログラム化します：モーダルを開き、フォームに記入し、状態を選び、送信し、そのアイデアがデータベースに登録されることを確認します。
信頼できるブラウザテストによって、修正しても UI が崩れないようにすることが目的です。

## UI をテスト可能にする

まず：用途を明確に表すセレクタを追加して、ボタンのテキストやスタイルに頼ったテストにならないようにします。
これには data 属性を利用します。

インデックスページでモーダルを表示させる部分です ― これに test フックをつけましょう：

```html
<!-- resources/views/ideas/index.blade.php -->
<button data-test="create-idea-button" x-data @click="$dispatch('open-modal', { name: 'create-idea' })">
  What's The Idea?
</button>
```

モーダル内の状態ボタンには、テストで使いやすいように属性値に状態名を含めましょう：

```html
<!-- モーダルのフォーム内部 -->
<button type="button" data-test="button-status-pending" @click="selected = 'pending'">Pending</button>
<button type="button" data-test="button-status-in_progress" @click="selected = 'in_progress'">In Progress</button>
<button type="button" data-test="button-status-completed" @click="selected = 'completed'">Completed</button>
```

そしてもちろんフォームの送信ボタンにも：

```html
<button type="submit" data-test="create-idea-submit">Create</button>
```

このような属性によって、後でラベル名を変えたりスタイルを変更したりしてもテストに影響が出ないようにします。

## ブラウザテストを作成する（正常系）

ログインしてモーダルを開き、フィールドに記入し、状態をクリックして送信し、データベースで結果を確認するというブラウザ／機能テストを作成しましょう。

基本的な流れはこのようになります（Pest を利用した場合の疑似コード）：

```php
// tests/Browser/CreateIdeaTest.php
public function test_creates_a_new_idea()
{
    $user = User::factory()->create();

    $this->actingAs($user)
         ->visitRoute('ideas.index')
         ->click('@create-idea-button') // data-test セレクタ
         ->type('Some Great Idea', 'title')
         ->click('@button-status-completed')
         ->type('This is an example description.', 'description')
         ->click('@create-idea-submit')
         ->assertRouteIs('ideas.index');

    // データベースの確認
    $this->assertDatabaseHas('ideas', [
        'title' => 'Some Great Idea',
        'status' => 'completed',
        'description' => 'This is an example description.',
        'user_id' => $user->id,
    ]);
}
```

補足：

- `@create-idea-button` は `data-test="create-idea-button"` に対応しています。
- `type('value', 'field-name')` で input に入力します（textarea への入力も同様です）。
- テストユーザでログインするには `actingAs(User::factory()->create())` とします。

自身の環境（Pest か artisan）に従ってテストを実行します：

```bash
# Pest
./vendor/bin/pest tests/Browser/CreateIdeaTest.php

# または artisan
php artisan test --filter=CreateIdeaTest
```

## ルートとコントローラ (Store アクション)

POST ルートと store アクションを用意しているか確認しましょう。
フォームリクエストでバリデーションを行う場合、store の正常系はこのようになります：

```php
// routes/web.php
Route::post('/ideas', [App\Http\Controllers\IdeaController::class, 'store'])
    ->name('ideas.store')
    ->middleware('auth');
```

```php
// app/Http/Controllers/IdeaController.php
public function store(\App\Http\Requests\StoreIdeaRequest $request)
{
    // ログインユーザに自動的に紐づけるためにリレーションを利用
    $idea = auth()->user()->ideas()->create($request->validated());

    return redirect()->route('ideas.index')->with('success', 'Idea created.');
}
```

`auth()->user()->ideas()->create(...)` を使って新しいアイデアをログインユーザに紐づけましょう。

## 基本的なバリデーション（フォームリクエスト）

StoreIdeaRequest に追加するバリデーションルールの例です ― タイトル必須、詳細はオプション、状態は必須で値の制限あり：

```php
// app/Http/Requests/StoreIdeaRequest.php
public function authorize()
{
    return true; // または必要に応じてポリシーでガード
}

public function rules()
{
    return [
        'title' => ['required', 'string', 'max:255'],
        'description' => ['nullable', 'string'],
        'status' => ['required', 'in:pending,in_progress,completed'],
    ];
}
```

フォームリクエストは、未認証または不正な送信をコントローラが実行される前にブロックします。
これでテストが予想しやすくなります。

## デバッグのコツ

テストが失敗したら、サーバが何を受信したかコントローラでダンプしてみましょう。

```php
dd($request->all());
```

よくある原因：

- ログインしていない（`auth` ミドルウェアが入っています）。
- 違ったセレクタを指定（`data-test` の名前が合っているか確かめましょう）。
- hidden のデータ（状態は Alpine によって hidden データとして送信されます）。

## モーダルをいつでも使用可能／リセット可能にする

キャンセルのフローで２つのことをしようと思います：モーダルを非表示にし内部状態をリセットします。
キャンセルボタンでモーダルを閉じるイベントを発生させましょう：

```html
<button type="button" @click="$dispatch('close-modal')" aria-label="Close modal">Cancel</button>
```

そしてモーダルコンポーネントでは、`close-modal` を受信すると `show = false` とセットした上でモーダル内部の Alpine の状態をクリアするようにしましょう。

## UX と並び順の調整

- モーダルを適切なサイズ（max-width と `max-height: 80vh` にして内部でスクロールできるようにしましょう。
- 登録したアイデアが上に表示されるようにアイデアは新しい順に並べましょう。

```php
$ideas = Idea::latest()->get();
```

## まとめ

- モーダルのトリガー、状態ボタン、送信ボタンには、テストしやすくなるように data 属性をつけましょう。
- モーダルを開き、フィールドに入力し、状態を選択し、送信し、データベースを確認できるブラウザテストを作成しましょう。
- store のルートとコントローラ、フォームリクエストを実装して、`auth` で保護しましょう。
- 選択した状態を送信できるよう、Alpine.js と hidden データを使いましょう。
- キャンセルボタンから `close-modal` イベントを発生させ、モーダルを非表示にしてリセットしましょう。

必要であれば、自身のプロジェクトでそのまま使用できる、最小限の完全な Blade コンポーネント、状態選択部分の Alpine のコード、それから完全なテストコードをお渡しできます。
