# [コントローラ](https://laracasts.com/series/laravel-from-scratch-2026/episodes/10)

リソース操作とコントローラについて話しましょう。
新たな章、新たなシャツ — 今回はリソースに対して行う７つの基本操作の整理、フォームをインラインに保つべき場合と、アプリが成長しても整理された状態を保つために全てをコントローラに移すべき場合についてです。

## ７つのリソース操作

結局のところ、リソースに対して必要なアクションは７つです：インデックス、表示、作成（新規リソース作成フォーム）、保存（新規リソースの永続化）、編集（編集フォーム）、更新（編集内容の永続化）、破棄（削除）。
小規模なプロジェクトでは、作成フォームをインデックスビューに直接配置することもあります（全く問題ありません）。
ただし、作成ビュー専用のルートとハンドラーを割り当てることも可能です。

クロージャとして実装する場合の一般的なルート構成例（簡易版）は以下の通りです：

```php
// routes/web.php
use Illuminate\Http\Request;

// すべてのアイデアの表示
Route::get('/ideas', function () { /* ... */ });

// アイデアの新規作成フォームの表示
Route::get('/ideas/create', function () { /* ... */ });

// 新しいアイデアの保存
Route::post('/ideas', function (Request $request) { /* ... */ });

// アイデアを1件表示
Route::get('/ideas/{idea}', function ($idea) { /* ... */ });

// 編集フォームの表示
Route::get('/ideas/{idea}/edit', function ($idea) { /* ... */ });

// 編集内容の保存
Route::patch('/ideas/{idea}', function ($idea) { /* ... */ });

// アイデアの削除
Route::delete('/ideas/{idea}', function ($idea) { /* ... */ });
```

インデックスページに作成フォームを設置する場合、７番目のアクションは暗黙的にカバーされます。
別個の作成ページを希望する場合は、GET /ideas/create ルートを追加し、そのフォームを POST /ideas（ストアアクション）に送信させることができます。
どちらのアプローチも有効です。UX に合う方を選択してください。

### 定義：コントローラ

> コントローラとは、ルートに応答できるあらゆるアクションを収容するコンテナとしての役割を持つ単なるクラスです。

コントローラを、ルートファイルに多数のクロージャを記述する代わりに、これら７つのアクションを整理して格納するコンテナと捉えてください。

## リソースフルなコントローラの自動作成

アプリが１つのリソースを超える規模になると、ほとんどの開発者はコントローラを好みます。
コントローラは関連するアクションを集約する場所です。
Artisan を使えば素早く作成できます。

以下は、リソースコントローラを生成しモデルにリンクするために私が通常実行するコマンドです：

```bash
php artisan make:controller IdeaController --resource --model=Idea
```

これにより、７つの便利なメソッドで構成された IdeaController が生成されます。
php artisan make:controller を実行しプロンプトに応答すると、同じ結果が得られます（学習のために対話的に入力することもあります）。

以下が得られる典型的なコントローラの骨格です：

```php
// app/Http/Controllers/IdeaController.php
namespace App\Http\Controllers;

use App\Models\Idea;
use Illuminate\Http\Request;

class IdeaController extends Controller
{
    public function index()     { /* アイデアのリスト表示 */ }
    public function create()    { /* アイデアの新規作成フォーム表示 */ }
    public function store(Request $request) { /* 新規アイデアの保存 */ }
    public function show(Idea $idea) { /* アイデアを１件表示 */ }
    public function edit(Idea $idea) { /* 編集フォームの表示 */ }
    public function update(Request $request, Idea $idea) { /* 編集の保存 */ }
    public function destroy(Idea $idea) { /* アイデアの削除 */ }
}
```

show/edit/update/destroy にある型ヒント付きの Idea に注目してください — これはルートモデルバインディングが自動的に処理しているものです。

## ルートとコントローラアクションの紐づけ

クロージャからコントローラメソッドへの切り替えは、ほとんどコピー＆ペースト作業です。
インラインクロージャの代わりに、ルートをコントローラのアクションに指定します：

```php
use App\Http\Controllers\IdeaController;

// 変更前: クロージャ
Route::get('/', function () {
    // ...
});

// 変更後: コントローラアクション
Route::get('/', [IdeaController::class, 'index']);
```

その他も同様です：

```php
Route::get('/ideas/create', [IdeaController::class, 'create']);
Route::post('/ideas', [IdeaController::class, 'store']);
Route::get('/ideas/{idea}', [IdeaController::class, 'show']);
Route::get('/ideas/{idea}/edit', [IdeaController::class, 'edit']);
Route::patch('/ideas/{idea}', [IdeaController::class, 'update']);
Route::delete('/ideas/{idea}', [IdeaController::class, 'destroy']);
```

簡潔に済ませたい場合は、Laravel が提供する `Route::resource(‘ideas’, IdeaController::class)` で自動登録できます。
これは便利な機能ですが、私は明示的な記述を好みます。
単一のファイルに全ルートを明確に列挙するためです。
数年経てば、ルートを探し回るより全ルートを一目で確認できるファイルの価値がわかるでしょう。

## 慣習とその重要性

リソース指向のパターンと RESTful 規約に従うことで、予測可能性が得られます。
エンドポイント /ideas は常に「アイデアのコレクション」を意味します。
/ideas への POST は常に「作成」、/ideas/{id} への PATCH は「更新」を意味します。
同様に、コントローラアクション名にも慣例があります：index、create、store、show、edit、update、destroy — 私は可能な限りこれらに準拠します。

正直に言うと：**98% のケースではカスタムアクション名を考案することはありません。**
非標準の動作が必要な場合、通常は別のコントローラを作成してリソース指向アクションのいずれかにマッピングするか、明確な名前のルートを追加します。
これらの規約に従うことは、他の人（あるいは将来の自分）がコードを読む際に大きな利益をもたらします。

## 最後に思うこと

コントローラはルートを整理し、関連する動作をグループ化します。
小規模なデモではフォームを index にインラインで保持し、本格的な開発では専用の作成ビューを用意できます。
Artisan でリソース対応のコントローラをスケルトン生成し、ルートをコントローラメソッドに紐付け、7つの標準アクション名に従いましょう — この規約が将来の時間と混乱を節約します。

次回は、この基盤をさらに構築し、コントローラアクションをもう少し具体化していきます。
