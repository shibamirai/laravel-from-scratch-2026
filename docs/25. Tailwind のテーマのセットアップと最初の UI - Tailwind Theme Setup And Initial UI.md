再利用可能なフォームコンポーネントを抽出し、登録/ログインフローを完成させます（バリデーションとパスワードハッシュ処理を含む）。

## 簡単なコードレビュー

修正を始める前にコード解析ツールを実行し（私は CodeRabbit を使います）、重要度の高い提案に目を通します。
自分では見逃しがちな細かなことを見つけてくれます：enum とキャストのミスマッチや、フォームリクエストの認証漏れ、属性の重複、CSSの問題などです。

私が守る絶対のルールがあります：コミット前にレビューアを実行し、適切な提案を適用します。
アプリにとっては必ずしも提案されたことすべてが正しいわけではありませんんが、見直す価値があるものに気づかせてくれます。

## Tailwind のテーマと CSS ユーティリティ

CSS 変数とユーティリティクラスを対応づける（Tailwind v4以上）という機能を使うと、テーマを簡単に調整できます。
resources/css/app.css で Tailwind をインポートし、各コンポーネントで参照できる CSS 変数を定義します。

idea アプリではこのようにしています ― app.css の先頭で Tailwind をインポートし、変数を定義しています。

```css
@tailwind base;
@tailwind components;
@tailwind utilities;

/* テーマ変数 */
:root{
  --color-bg: 10 10 10;           /* rgb コンポーネント：rgb(var(--color-bg) / var(--tw-bg-opacity)) のように使用 */
  --color-foreground: 245 245 245;
  --color-card: 20 20 20;
  --color-border: 60 60 60;

  --color-primary: 34 197 94;     /* 緑系の色 */
}
```

これにより、text-primary, bg-card, border-border といったユーティリティクラスを作ることができます。
コンポーネントのクラスでこれらの変数を使用すれば、アプリ全体で一貫した見た目を整えるのに便利です。

また、一般的なコンポーネントのスタイル（ボタン、フォームフィールド）をコンポーネントの CSS ファイルとして取り出します。
ボタンコンポーネントの例（一部略）：

```css
/* components/_button.css (app.css でレイヤーとしてインポートされる) */
.btn {
  @apply inline-flex items-center rounded-md px-4 py-2 text-sm font-medium;
  background-color: rgb(var(--color-primary));
  color: rgb(var(--color-foreground));
}

.btn-outline {
  @apply bg-transparent border;
  border-color: rgb(var(--color-border));
  color: rgb(var(--color-primary));
}

.btn-ghost {
  @apply bg-transparent;
}
```

これらを components レイヤーの下に配置しておくと、ユーティリティが見つけてくれます。

```css
@layer components {
  /* .btn, .btn-outline, などをここでインポート */
}
```

それから Vite/Vite のサーバを実行し（vite, npm run dev, または laravel-vite）、ホットリロードさせます。

## レイアウト、ナビゲーション、再利用可能なフォームコンポーネント

レイアウトの HTML をコピーペーストしてまわるのではなく、レイアウトコンポーネントとして取り出して、ナビゲーションや中心寄せのコンテンツエリアを提供します。

<p>layouts/idea.blade.php (イメージ):</p>

```html
<nav class="border-b border-border">
  <div class="max-w-6xl mx-auto flex items-center justify-between h-14 px-4">
    <a href="/" class="flex items-center">
      <img src="/images/logo.png" alt="Idea logo" width="100" />
    </a>

    <div class="flex items-center gap-5">
      <a href="{{ route('login') }}" class="text-sm">Log In</a>
      <a href="{{ route('register') }}" class="btn">Register</a>
    </div>
  </div>
</nav>

<main class="max-w-6xl mx-auto px-4 py-12">
  {{ $slot }}
</main>
```

フォーム用のコンポーネント群を作りましょう。
私がよく使うのは：

- フォーム全体を囲むコンポーネント (x-forms.form) ：センター寄せレイアウトでタイトルと詳細を受け取る
- 名前、ラベル、型、追加する属性を受け取るフォームフィールドコンポーネント (x-forms.field)
- CSS コンポーネント (btn, input, textarea) クラスを用意し、ユーティリティクラスの繰り返しを避ける

登録ビューでの使用例：

```html
<x-forms.form title="Register An Account" description="Start tracking your ideas today.">
  <x-forms.field name="name" label="Your Name" />
  <x-forms.field name="email" label="Your Email" type="email" />
  <x-forms.field name="password" label="Password" type="password" />
  <button type="submit" class="btn mt-4">Create Account</button>
</x-forms.form>
```

フォームフィールドコンポーネント（簡略化しています）：

```html
@props(['name','label','type' => 'text'])
<label class="block text-sm font-medium">{{ $label }}</label>
<input
  type="{{ $type }}"
  name="{{ $name }}"
  value="{{ old($name) }}"
  {{ $attributes->merge(['class' => 'input w-full mt-1']) }}
/>
<x-forms.error name="{{ $name }}" />
```

ここにある `old($name)` は重要です：バリデーションエラーで Laravel が元のページにリダイレクトしたとき、ユーザが一から再入力しなくて済むように、前の入力値を保持します。

## 信頼できる UI テストのための data-test 属性

様々な場所（ナビゲーションリンク、フォームボタン）にある似たような要素の１つをクリックするようなテストでは、ターゲット要素にデータ属性をつけましょう。
私はよく `data-test` という属性を使ってテストを明確で信頼できるものにします。

<p>例：</p>

```html
<button data-test="create-idea-button">Create Idea</button>
<button data-test="button-status-completed">Completed</button>
<button data-test="create-idea-submit">Create</button>
```

テストの側では `@` という簡易表現 (Pest/Playwright で使用されるラッパー) を使って、`@create-idea-button` と `[data-test="create-idea-button"]` を紐づけます。
こうすることでテストが分かりやすくなり、いい加減さがなくなります。

## ルートと認証ビュー

基本的なユーザ登録とログインのルートを作成し、それらをコントローラに紐づけてビューを返すようにします：

```php
use App\Http\Controllers\Auth\RegisteredUserController;
use App\Http\Controllers\Auth\SessionsController;

Route::get('/register', [RegisteredUserController::class, 'create'])->name('register');
Route::post('/register', [RegisteredUserController::class, 'store']);

Route::get('/login', [SessionsController::class, 'create'])->name('login');
Route::post('/login', [SessionsController::class, 'store']);

Route::post('/logout', [SessionsController::class, 'destroy'])->name('logout');
```

ユーザ登録とログインのビューでは先ほどつくったフォームコンポーネントを使用します。
ナビゲーションバーでは Blade の `@auth` と `@guest` ディレクティブを使って認証時用／ゲスト用のリンクを表示します。

## コントローラの例とバリデーション

リクエストはコントローラでインラインでバリデーションします（後でフォームリクエストに取り出すこともできます）。
ユーザ登録のアクションの例です（一部略）：

```php
use Illuminate\Http\Request;
use App\Models\User;
use Illuminate\Support\Facades\Auth;

public function store(Request $request)
{
    $attrs = $request->validate([
        'name' => ['required','string','max:255'],
        'email' => ['required','email','max:255','unique:users,email'],
        'password' => ['required','string','min:8'],
    ]);

    $user = User::create([
        'name' => $attrs['name'],
        'email' => $attrs['email'],
        'password' => $attrs['password'], // キャストが用意されていれば Laravel がハッシュ化を行う（次のセクション参照）
    ]);

    Auth::login($user);

    return redirect()->route('ideas.index')->with('success', 'Welcome!');
}
```

続いてセッション保存（ログイン）アクションです：

```php
public function store(Request $request)
{
    $credentials = $request->validate([
        'email' => 'required|email',
        'password' => 'required|string',
    ]);

    if (! Auth::attempt($credentials)) {
        return back()->withErrors(['password' => 'The provided credentials do not match our records.'])
                     ->onlyInput('email');
    }

    $request->session()->regenerate();

    return redirect()->intended(route('ideas.index'))->with('success', 'You are now logged in.');
}
```

ログインに失敗したら一般的なエラーメッセージを表示します（セキュリティのためメールアドレスが存在するかどうかを知らせてはいけません）。
セッション固定化のリスクを減らすため、成功時には `session()->regenerate()` も実行します。

## パスワードのハッシュ化とモデルのデフォルト値

Laravel 12 ではパスワードのハッシュ化が簡単になりました：ユーザモデルでパスワードを `hashed` にキャストするだけで、Hash::make を呼び出す必要がなくなりました。

```php
// app/Models/User.php
protected $casts = [
    'password' => 'hashed',
];
```

`User::create(['password' => 'secret'])` のようにしてユーザを作成すると、Laravel が自動でハッシュ化してくれます。

アイデアのようなモデルでは、デフォルト値を与えてガードを無効化しておくと便利です（マスアサインメントには注意してください）：

```php
// app/Models/Idea.php
class Idea extends Model
{
    protected $attributes = [
        'state' => 'pending',
    ];

    protected $guarded = []; // 簡便化のため；大規模なアプリでは $fillable を使用した方がよい
}
```

デフォルト値をセットして、テストやシーダでいちいち `state` を指定する手間を省きます。

> マイグレーションはデータベースのバージョン管理です。

マイグレーションはチームに分かりやすいようにスキーマを発展させる方法です。
こうやってマイグレーションを作成してください：

```bash
php artisan make:migration create_ideas_table
```

マイグレーション内の up() でカラムを定義します；down() で変更をリセットします。
開発途中でデータがなくなっても構わないのであれば、`php artisan migrate:refresh` によってすべてのマイグレーションを一からやり直せます。
チーム開発や運用でのワークフローでは追加的マイグレーション（カラムを追加／削除するマイグレーションを作成して `php artisan migrate` を実行）の方がいいでしょう。

## ビューでバリデーションのフィードバックを行う

バリデーションに失敗すると、Laravel はセッションにエラーを送って元の画面にリダイレクトします。
入力箇所の下にインラインでエラーを表示しましょう。

Blade ディレクティブかちょっとしたコンポーネントを使います：

書き方：

```html
@if ($errors->has('description'))
  <p class="text-red-500 text-sm">{{ $errors->first('description') }}</p>
@endif
```

短縮した書き方：

```html
@error('description')
  <p class="text-red-500 text-sm">{{ $message }}</p>
@enderror
```

もしくは一貫した書き方をするために x-forms.error コンポーネントとして取り出します。

バリデーションエラー時に入力値が保存されているように、input で `old('field')` も使用しましょう。

```html
<input name="email" value="{{ old('email') }}" />
```

## UX とアクセシビリティに関する注意点

- アイコンだけのボタンには `aria-label` を付与しましょう
- 意味を持ったマークアップ（nav, main, header）を使用しましょう
- ロゴには `alt` で説明をつけましょう
- ユーザ登録／ログインのリンクは `@guest` の下に、ログアウトフォームは `@auth` の下に配置しましょう
