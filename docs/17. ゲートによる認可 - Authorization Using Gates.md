# ゲートによる認可

認可について話しましょう ― ゲートの紹介、Blade の @can ディレクティブ、ルートレベルでの保護、それから実際に使用できるいくつかの実用的なパターンです。

## サービスプロバイダーとルールの登録場所

ビューやルートで認可を行う前に、どこかでそのルールを登録する必要があります。
わたしはこれをサービスプロバイダーで行います。

> サービスプロバイダーとはアプリケーションの自動起動コマンドのようなものです。

サービスプロバイダーでは、何かしらをフレームワークに登録し、サービスをコンテナーにまとめ、アプリの起動シーケンスに組み込みます。

簡単な例：管理者エリアを閲覧できるユーザーかどうかを判定する view-admin という名前の Gate を登録する。

```php
// app/Providers/AppServiceProvider.php

use Illuminate\Support\Facades\Gate;
use Illuminate\Auth\Access\Response;
use App\Models\User;

public function boot()
{
    Gate::define('view-admin', function (?User $user) {
        // ブール値かアクセス応答を返す
        return $user && $user->id === 1;
    });
}
```

ゲストユーザーでも判定できるルールにする場合は、上記のように引数を null 許可してください。
ログインユーザーだけに限る場合は、型の User につけている `?` を取り除きます。

## Blade の @can ディレクティブの使用

Blade はゲートの条件に応じて HTML をレンダリングする便利なディレクティブを提供します。
これは認可チェックの構文糖衣です。

```html
@can('view-admin')
  <a href="{{ route('admin.index') }}">Administrator</a>
@endcan
```

裏では先ほど登録したゲートクロージャーと同じものが実行されます。
注意点：デフォルトでは、ゲートは認証済みユーザーを必要とします。
訪問者がログインしていない場合、ゲートは実行されず、@can 関数は false を返します。
ゲストユーザーでも実行させたい場合は、ゲートクロージャでユーザーが null でも受け取れるようにし（先の例のように）、null のケースに対応させます。

## ルートレベルでの認可

URL を知られていたとしても許可されていないユーザがそのエンドポイントにアクセスできないように、ルート自体を保護することができます。
ルートの `can` メソッドを使用します（ゲートと同じ名前を指定します）。

```php
Route::get('/admin', function () {
    return 'Admin only';
})->can('view-admin');
```

これはミドルウェアと同じような働きをします：ゲートはルートの前に実行されます。
ゲートがアクセスを拒否するとユーザーは 403 (またはあなたが選んだレスポンス) を受け取ります。

## コントローラーまたはクロージャーでの認可

もし認可チェックをコントローラー（またはルートクロージャ）内で行いたければ、Gate::authorize を使いましょう：これでアクセスが拒否されると AuthorizationException 例外が発生し、それを受けて Laravel が適切な応答を返します。

```php
use Illuminate\Support\Facades\Gate;

public function index()
{
    Gate::authorize('view-admin');

    // もう管理者用の情報を見せても安全です
}
```

これはルートレベルで `can` 制約を適用するのと同じです：違いはチェックする場所だけです。
どちらが好むかはチームによって異なります ― ルールがシンプルならルートレベルがきれいですし、たくさんのロジックがあるときはコントローラーレベルの方が明確で分かりやすいです。

## ゲートのレスポンスのカスタマイズ

ゲートはブール値の他、アクセスレスポンスを返すことができます。
レスポンスを返すことで HTTP ステータスを制御することができます。
もしエンドポイントの存在を隠したければ、"not found" の代わりに "forbidden" を返すことができます。

```php
use Illuminate\Auth\Access\Response;

Gate::define('view-admin', function (?User $user) {
    if ($user && $user->id === 1) {
        return Response::allow();
    }

    // 403 の代わりに 404 を返す
    return Response::denyAsNotFound();
});
```

つまり、管理者権限を持たないユーザーが `/admin` にアクセスした場合、セキュリティポリシーとして 403 ではなく 404 を返すことができます。

## シンプルなユーザーの役割（ロール）チェック

小さなアプリでチェックを行う最も簡単な方法は、ユーザーにちょっとしたヘルパーを用意し、ゲートでそれを確認することです。
ユーザーモデルに isAdmin という小さなメソッドを追加します：

```php
class User extends Authenticatable
{
    public function isAdmin(): bool
    {
        return $this->role === 'admin';
    }
}
```

するとゲートはシンプルで分かりやすくなります：

```php
Gate::define('view-admin', function (User $user) {
    return $user->isAdmin();
});
```

もしロールカラムをまだ用意していなければ、これはただのおもちゃのアプリなので最初は `id === 1` かどうかをチェックするだけで十分です ― ですが、ユーザーやチームメイトが増えてきたらロールフィールドを追加しましょう。

## 実践的なコツをいくつか

ログインしていない訪問者についてもチェックを行いたい場合は、ユーザーパラメータを null 可にし、`null` の場合の処理を追加してください。
エンドポイントを隠したい場合は、ゲートから not-found レスポンスを返します。
チーム内での分かりやすさを基準にして、認可をルート、コントローラー、ビューのどこで行うかを決めてください ― 大事なのは”正しさ”より一貫性です。
